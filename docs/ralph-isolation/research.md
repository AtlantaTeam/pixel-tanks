## Канареечная проверка границы (фаза 3, #184)

**Что это.** Скрипт `scripts/secret-canary.mjs` пытается достать секреты петли всеми
известными каналами и печатает воспроизводимый отчёт, какие из них сейчас открыты. Смысл
— сделать «изолировано» проверяемым фактом: без канарейки «мы почистили env» нельзя
отличить от «токен пришёл другим путём».

**Как запускать.**

```bash
npm run security:canary:baseline   # = node scripts/secret-canary.mjs
```

Скрипт read-only, значений секретов не печатает (только факт наличия и длину), всегда
завершается с кодом 0 — на этой фазе это **измерение, а не красный гейт**. В гейт
канарейка НЕ встроена: до env-санации (фаза 4, #190) это был бы вечно-зелёный чек,
который находит секреты и молчит, — против fail-closed-духа обоих PRD. Обязательным
красным чеком канарейка становится в фазе 4.

**Какие каналы проверяются.**

- **env-переменные:** `GH_TOKEN`, `GITHUB_TOKEN`, `CLAUDE_CODE_OAUTH_TOKEN`,
  `ANTHROPIC_API_KEY`, `TG_BOT_TOKEN`, `RALPH_TG_BOT_TOKEN`. Помимо трёх названных в issue
  секретов канарейка ищет их алиасы по семействам (`GH_*`/`CLAUDE_*`/`ANTHROPIC*`/`TG_*`):
  санация делается allowlist-ом, поэтому проверять надо ШИРЕ, чем чистим, иначе «почистили
  `GH_TOKEN`» не отличить от «утёк `GITHUB_TOKEN`».
- **файлы:** `~/.config/gh/hosts.yml` (токен `gh` CLI), `~/.claude/.credentials.json`
  (токен `claude` CLI), `/root/ralph.env` (env-файл секретов раннера).

**Отчёт воспроизводим:** тот же вход (env + файлы) → тот же текст. Пример формата
(**числа условные, иллюстрируют формат строк**, а не реальный вывод — фактический снимок
и длины токенов зафиксированы в `scripts/secret-canary.baseline.json`, #187; счётчик в
заголовке `открытые/всего` меняется при расширении списков каналов):

```
Канарейка секретов — открытые каналы к секретам петли: <открытых>/<всего>
  🔓 ОТКРЫТ  env:GH_TOKEN: <найден, длина …>
  🔒 закрыт  env:GITHUB_TOKEN: <пусто>
  🔒 закрыт  file:~/.config/gh/hosts.yml (gh CLI OAuth-токен): <нет файла>
  🔓 ОТКРЫТ  file:/root/ralph.env (env-файл секретов раннера): <читается, найден маркер "GH_TOKEN", длина …>
```

Логику детекта (env/файлы, раскрытие `~`, редакция значений) покрывает обычный
vitest-тест `scripts/secret-canary.test.js` на инжектированных фикстурах — он гоняется в
гейте зелёным и НЕ читает настоящие секреты.

## Источник токена `gh` на VDS (фаза 3, #186)

**Факт.** На боевой VDS `gh` CLI берёт токен из **env-переменной `GH_TOKEN`**, не из
файла: `gh auth status` явно называет источник —
`✓ Logged in to github.com account Pelmenya (GH_TOKEN)`. Файл `~/.config/gh/hosts.yml`
на этой машине вообще отсутствует — файловый канал (`gh auth login` с сохранением в
конфиг) здесь никогда не использовался, только env.

**Как токен попадает в env.** По `RUNBOOK.md` раннер стартует командой
`set -a && . /root/ralph.env && set +a && node ralph.js ...` — `GH_TOKEN` объявлен в
`/root/ralph.env` (root-only, `0600`) и наследуется процессом `ralph.js` и всеми его
дочерними процессами (`gh`, `git`, чеки гейта) как обычная переменная окружения.
Отдельного механизма (systemd `EnvironmentFile`, сохранение токена в OS keyring) не
обнаружено — только shell-source перед стартом.

**Следствие для фазы 4 (env-санация).** Allowlist должен явно пропускать `GH_TOKEN`
только тем шагам, которым он нужен (push/PR/мердж), и вычищать его из env чеков гейта
(`npm run test`, e2e, `npm ci`) — сегодня он у них есть только потому, что весь цикл
живёт в одном процессе. Файловый канал (`hosts.yml`) на этой машине сейчас пуст, но
канарейка обязана проверять его наравне с env: на другой машине или после
`gh auth login` состояние легко изменится, а проверка не должна зависеть от текущего
факта одной VDS.

**Второй критерий готовности уже выполнен существующим кодом.** `scripts/secret-canary.mjs`
уже включает `GH_TOKEN` в `SECRET_ENV_VARS` и `~/.config/gh/hosts.yml` в
`SECRET_FILE_CHANNELS` (см. «Канареечная проверка границы» выше, #184) — канарейка
проверяет оба канала без дополнительных изменений.

## Фикстура npm-зависимости с lifecycle-скриптом (фаза 3, #185)

**Что это.** `scripts/canary-postinstall-fixture/` — минимальный npm-пакет с `postinstall`,
эмулирующий скомпрометированную зависимость: при установке его lifecycle-скрипт применяет
ТОТ ЖЕ канареечный подход, что и `secret-canary.mjs` (env-переменные + известные файловые
пути токенов), и печатает воспроизводимый отчёт, какие каналы к секретам петли открыты
изнутри postinstall. Фикстура **не** зависимость приложения (в `package.json` проекта её
нет) — иначе её postinstall гонялся бы при каждом `npm ci` гейта и на фазе 3 это был бы
вечно-зелёный чек, находящий секреты и молчащий (тот же аргумент, что и для канарейки).

**Как запускать.**

```bash
npm run canary:postinstall   # = node scripts/canary-postinstall.mjs
```

Скрипт ставит фикстуру как настоящую зависимость во временный проект и прогоняет `npm ci`.
postinstall всегда завершается `exit 0` — критерий «фикстура не ломает обычный `npm ci` на
честном коде»: это измерение канала, а не красный чек. Значений секретов не печатает.

**Ключевой факт npm 11 (установлен здесь).** `npm 11.16` по умолчанию НЕ исполняет
install-скрипты ЗАВИСИМОСТЕЙ — новый supply-chain-гейт `allow-scripts`: незнакомый пакет с
`postinstall` даёт лишь `npm warn allow-scripts ... not yet covered by allowScripts`, сам
скрипт не запускается. Значит **плоский `npm ci` гейта** (`ralph.js`: `execSync('npm ci')`,
без флагов скриптов) **уже сам блокирует postinstall скомпрометированной зависимости** —
это факт-митигация npm, не наша заслуга. Скрипт зависимости исполняется только при явном
`--dangerously-allow-all-scripts` (или allowlist пакета, или на старом npm). Демо поэтому
делает два прогона: (1) как в гейте — postinstall зависимости заблокирован; (2) с
`--dangerously-allow-all-scripts --foreground-scripts` — baseline канала, что postinstall
увидел бы, будь скрипты разрешены.

Следствия для фаз 4–5:

- Env-санацию (фаза 4) всё равно делаем: она закрывает канал для того, что npm 11 НЕ
  блокирует — install-скрипты САМОГО проекта (root-пакет; его `postinstall` npm гоняет по
  умолчанию — проверено) и код ТЕСТОВ гейта (`npm run test`/e2e исполняются с полным env).
  Блокировка dep-скриптов сужает вектор, но не закрывает вектор «код гейта видит секреты».
- Опция усилить митигацию: гонять `npm ci` гейта с явным `--ignore-scripts` — отдельное
  решение (может сломать легитимные build-скрипты, напр. `sharp`), в скоуп #185 не входит,
  зафиксировано как кандидат для фазы 4.

**Baseline канала (до санации).** Прогон демо на dev-машине ожидаемо находит открытыми:
env-токены раннера (`GH_TOKEN`, `RALPH_TG_BOT_TOKEN`), `~/.claude/.credentials.json`,
`/root/ralph.env`. Прогон на боевой VDS и фиксация вывода — #187 (тем же скриптом).

**Почему детект дублирует `secret-canary.mjs`, а не импортирует.** Настоящая
скомпрометированная зависимость самодостаточна: npm копирует её в `node_modules`, импорт
`../secret-canary.mjs` (за пределы каталога пакета) после копирования сломался бы и уронил
postinstall ненулевым кодом — это нарушило бы критерий «не ломает `npm ci`». Поэтому
`probe.mjs` не тянет ничего за пределы своего каталога. Цена — два списка секрет-каналов;
их синхронность держит тест-барьер `scripts/canary-postinstall-fixture.test.js` (сверяет
`SECRET_ENV_VARS`/`SECRET_FILE_CHANNELS` с `secret-canary.mjs`): добавили секрет в канарейку,
забыли в фикстуре — тест краснеет.

## Остаточный риск env-санации (фаза 4, #192)

**Что это.** Env-санация (вариант (а) из PRD) закрывает канал через переменные окружения, но
файловый канал (домашний каталог того же пользователя ОС) остаётся открытым до варианта
(б)/(в) — отдельного пользователя ОС или контейнера.

**Конкретные каналы.**

- `~/.config/gh/hosts.yml` (OAuth-токен gh CLI) — на боевой VDS сегодня отсутствует
  (фактический источник токена — env-переменная `GH_TOKEN`, см. раздел «Источник токена
  `gh`»), но канарейка проверяет его наравне с env: на другой машине или после `gh auth login`
  состояние может измениться.
- `~/.claude/.credentials.json` (токен claude CLI) — существует на боевой машине, содержит
  актуальный токен (проверено в базовом запуске канарейки, #187). Процесс гейта (раннер и
  его чеки) запускаются от пользователя root, могут читать этот файл.
- `/root/ralph.env` (env-файл секретов раннера) — содержит `GH_TOKEN`, `RALPH_TG_BOT_TOKEN`,
  `RALPH_TG_CHAT_ID`; находится в каталоге root, читается кодом гейта.

**Почему не закрывается в (а).** Env-санация работает на уровне процесса раннера —
чеки гейта (тесты, `npm ci`, e2e) запускаются как дочерние процессы этого процесса и
наследуют его uid/gid (root). Доступ к файлам — по правам ОС, санация окружения это не
меняет. Закрытие файловых каналов требует жёсткой границы: отдельный пользователь ОС
(вариант б, т.к. разные uid → разные домашние каталоги) или namespace/контейнер
(вариант в).

**Следствие.** (а) — **не песочница** в смысле полной изоляции от секретов. Санация
закрывает РОВНО env-канал, файловые каналы остаются открытыми:

- **Env-канал закрыт.** Значения секретов не видны в `process.env` дочерних процессов
  чеков — allowlist их вырезает (это и проверяет канарейка: env-секрет в `leaked` = красный).
- **Файловые каналы открыты, лишь наблюдаются.** Чеки бегут тем же uid (root), поэтому
  `~/.config/gh/hosts.yml`, `~/.claude/.credentials.json`, `/root/ralph.env` доступны по
  правам ОС — санация env их НЕ закрывает и НЕ блокирует чтение. Канарейка их только
  **детектирует** (для принятых #192 — как остаточный риск, не краснит; см. ниже).

Практический вывод: PR, дружащий с секретом, **не достанет его из env**, но при желании
**прочитает содержимое читаемого файла** (`~/.claude/.credentials.json`, `/root/ralph.env`)
и извлечёт токен оттуда — а через него использует `gh`/`claude` CLI. Именно поэтому (а) —
не песочница: файловый канал закрывает только жёсткая граница (б)/(в).

**Мониторинг и закрытие.** Канарейка (`secret-canary-gate.mjs`) в гейте **логирует**
открытые файловые каналы, но краснит по ним ИЗБИРАТЕЛЬНО: принятые остаточные каналы #192
(`~/.claude/.credentials.json`, `/root/ralph.env`) отмечаются в выводе чека и гейт не
краснят; **непринятый** файловый канал (например, `~/.config/gh/hosts.yml` после
`gh auth login` — на VDS его сегодня нет) в `RESIDUAL_RISK_CHANNELS` отсутствует и даёт
**красный** — ровно как любой открытый env-канал (см. `RESIDUAL_RISK_CHANNELS`,
`evaluateGateVerdict`). Это детектирование пост-фактум (наблюдение, не предотвращение).
Выбор границы (б)/(в) для полной изоляции — **issue #196**, фаза 6.

## Санация не ломает толстые prod-чеки (фаза 4, #191)

**Что проверялось.** Критерий готовности фазы 4: e2e сохраняет кэш браузеров Playwright и
поднимает свой dev-сервер под санированным env; полный гейт (security, coverage, e2e)
зелёный на эталонном коммите ПОСЛЕ включения санации (#188/#189/#190, уже в `origin/main`).

**Как проверяли.** Прогнали ровно ту последовательность, что гоняет `checksGreen` в
prod-профиле, тем же `buildSanitizedGateEnv()` — build → security:canary → security:audit →
test:coverage → `CI=1 npm run test:e2e` — на коммите `c056e70` (голова ветки на момент
проверки, все три issue фазы 4 включены). Санированный env на машине проверки — 9 переменных
(`PATH, HOME, USER, LOGNAME, SHELL, PWD, LANG, COLORTERM, TERM`); `CI`/`PLAYWRIGHT_BROWSERS_PATH`
не осели в исходном `process.env` этой машины, но при простановке `CI=1` (как это делает
prod-гейт) остаются пропущены дальше — allowlist их не режет.

> **Прокси-переменные egress (ревью #247).** Под санированным env в сеть ходят `npm ci`
> (`syncDepsIfLockChanged`, `ensureRunnerWorktree`) и `security:audit` (`npm audit`). На
> машине проверки прокси-переменных в `process.env` не было (egress прямой), поэтому
> зелёный audit их не задействовал. Чтобы боевая VDS не упала «инфраструктурой», если её
> egress вдруг настроен через env-прокси, `HTTP_PROXY`/`HTTPS_PROXY`/`NO_PROXY` (оба
> регистра) добавлены в allowlist явной записью. `npm_config_*` СОЗНАТЕЛЬНО НЕ пропускаем
> (семейство несёт токены реестра — это вернуло бы секрет-канал).

**Результат — всё зелёное:**

- `build` — `next build` (Turbopack) собрался под 9-переменным env без обращения к сети.
- `security:canary` (`secret-canary-gate.mjs`) — 0 открытых env-каналов из 9 проверяемых;
  два файловых канала (`~/.claude/.credentials.json`, `/root/ralph.env`) отмечены как
  остаточный риск (#192), не красят чек — ожидаемое поведение.
- `security:audit` — новых advisory нет, только 8 известных baseline.
- `test:coverage` — полный прогон `vitest run --coverage` прошёл под санированным env.
- `test:e2e` (`CI=1 npm run test:e2e`) — 6/6 тестов зелёные (клавиатура, тач, мобильные
  вьюпорты), 1.8 мин. Playwright нашёл закэшированные браузеры через `HOME` (дефолтный
  путь `~/.cache/ms-playwright`, отдельный `PLAYWRIGHT_BROWSERS_PATH` на этой машине не
  задан — но переменная в allowlist на случай, если он задан на VDS) и поднял свой
  `next start --port 3051` с `reuseExistingServer: false`, как задумано в
  `playwright.config.ts` для CI-режима.

**Побочное наблюдение (не блокер, не относится к санации).** `next start` печатает
предупреждение `"next start" does not work with "output: standalone" configuration. Use
"node .next/standalone/server.js" instead` — сервер всё равно поднимается и e2e проходит,
но это готовый к порче угол (следующая major-версия Next может сделать это фатальным, а не
предупреждением). Существовало и до санации (не вызвано ей) — заведён отдельным issue вне
скоупа #191, не в milestone текущей фазы.

**Регрессия закрыта тестом, не только измерением.** `gate-env.test.js` теперь пинит
присутствие `CI` и `PLAYWRIGHT_BROWSERS_PATH` в `gate-env-allowlist.json` — случайное
удаление любой из них при будущей правке allowlist красит `gate-env.test.js` сразу, не
дожидаясь падения дорогого e2e-чека в самом гейте.

## Обоснования оставленных вызовов `sh()` (фаза 5, #194)

**Контекст.** Фаза 5 требует перевести мутации гейта **на пути к автодеплою**
(`gh pr merge`, `git fetch`/`checkout` в гейте) на `execFileSync` с argv (#193 уже
сделано). Остаток вызовов `sh()`/`shFn` (см. полную таблицу в разделе «Итого» ниже)
остаётся на строке через шелл — дыры нет (значения из конфига/API квотируются `shq()`),
но класс риска остаётся (RCE-канал, если значение однажды придёт без `shq`). Полный
рефактор `sh()` — отдельное решение (#252); эта фаза зафиксирует обоснование для
каждого оставшегося вызова. **Важно:** «мутации гейта переведены на argv» относится
ТОЛЬКО к семье автодеплоя (merge + fetch/checkout детача PR-головы). Мутации меток
(`removeBlockedLabel`/`addBlockedLabel`) и фетчи хореографии (`refreshRunnerWorktree`,
`ensureRunnerWorktree`, `phaseDiffFiles`) — тоже мутации, но НЕ на пути к автодеплою;
они разобраны ниже и остаются кандидатами #252.

### Вызовы `sh()` в `ralph.js`

#### `ghJson()` — читаемые gh-команды с авторетраем

**Что это.** Функция `ghJson(cmd, attempts = 3)` выполняет gh-команды (чтение) с
автоматическим повтором и backoff при транзиентных сетевых сбоях.

```javascript
function ghJson(cmd, attempts = 3) {
    let lastErr;
    for (let i = 1; i <= attempts; i++) {
        try {
            return JSON.parse(sh(cmd));
        } catch (e) {
            lastErr = e;
            if (i < attempts) {
                log(
                    `⚠ gh-чтение не удалось (попытка ${i}/${attempts}): ... повтор через ${5 * i}с`,
                );
                sleep(5000 * i);
            }
        }
    }
    throw lastErr;
}
```

**Использование:** чтение PRs, issues, миллестоунов в цикле сдачи фазы
(`sweepClosedMilestones`, `openPhasePRs`, `checkMilestoneExists` и пр.).

**Обоснование остаться на `sh()`:**

1. **Только чтения, идемпотентны.** Команды — queries к GitHub API (gh issue list, gh api GET),
   не мутации. Это НЕ значит, что shell-injection-вектора нет: инъекция в строку
   `gh issue list …` исполнила бы произвольную команду независимо от того, что сама
   gh-операция read-only (инвариант 7 CLAUDE.md ralph прямо называет новый `sh()` с
   внешним значением без `shq` RCE-каналом). Argv устранил бы этот класс структурно;
   сейчас он оплачен квотированием (см. п. 2) — трейдофф, а не отсутствие риска. Довод
   «read-only» относится к идемпотентности (повтор безопасен), не к безопасности строки.

2. **Параметры защищены.** Переменные (milestone, branch, PR-число), попадающие в cmd,
   квотируются через `shq()` перед передачей в ghJson — это достаточно для чтений. Пример:

    ```javascript
    ghJson(`gh issue list --milestone ${shq(milestone)} --state open ...`);
    ```

3. **Ретрай — критичная фича.** Ночные прогоны без человека часами. Транзиентный сетевой сбой
   (timeout, rate-limit) не должен убивать сессию. Логика ретрая с экспоненциальным backoff
   (5/10/15 сек) в `ghJson` — нельзя потерять.

4. **Не на пути мутаций.** `ghJson` — наблюдение за состоянием PR/issues, не управление.
   Мутации гейта **на пути к автодеплою** (`gh pr merge`, `git fetch`/`checkout` детача
   PR-головы) переведены на argv в #193; прочие мутации (метки, фетчи хореографии) —
   см. таблицу «Итого» и #252.

### Закрытие milestone (`sweepClosedMilestones`/`closeMilestoneByTitle`) — побочка после мерджа

**Что это.** Два места закрывают milestone после успешного мерджа:

```javascript
// в sweepClosedMilestones():
sh(`gh api -X PATCH repos/{owner}/{repo}/milestones/${shq(ms.number)} -f state=closed`);

// в closeMilestoneByTitle():
sh(`gh api -X PATCH repos/{owner}/{repo}/milestones/${shq(ms.number)} -f state=closed`);
```

Обе используют `gh api -X PATCH` (мутация) через sh().

**Обоснование остаться на `sh()`:**

1. **Не в гейте, не на пути к автодеплою.** Закрытие milestone — побочка post-merge,
   происходит после успешного `gh pr merge` (уже на argv). Не критична для гейта/мерджа.

2. **Fail-open по дизайну.** Если закрытие упадёт, сессия не ломается — логируется
   предупреждение, milestone закроет sweep на следующем старте. Нет fail-closed требования.

3. **Параметры защищены.** Номер milestone берётся из API-ответа (integer), квотируется
   через shq(). Нет пользовательского ввода.

4. **Полный рефактор мутаций — отдельное решение.** PRD требует argv для мутаций на пути
   к автодеплою. Побочные мутации (sweep, уборка) — кандидаты для фазы N+1 (**#252**),
   не блокируют фазу 5.

### Мутации меток blocked (`removeBlockedLabel`/`addBlockedLabel`) — путь решения гейта, но не автодеплоя

**Что это.** Раннер сам снимает и возвращает label `blocked` в цикле разбора blocked
(#217/#223), через `shFn` (дефолт — `sh`):

```javascript
shFn(`gh pr edit ${shq(num)} --remove-label blocked`); // removeBlockedLabel
shFn(`gh pr edit ${shq(num)} --add-label blocked`); // addBlockedLabel
```

**Обоснование остаться на `sh()` (пока):**

1. **Мутация, но НЕ на пути к автодеплою.** Метка влияет на ВЕРДИКТ гейта в цикле разбора
   blocked (#217) — это честная мутация на пути принятия решения, поэтому она вынесена
   отдельным пунктом, а не спрятана в «читающие». Но в `main` она ничего не двигает: сам
   мердж (автодеплой) закрывает `gh pr merge`, уже на argv. Класс риска ниже, чем у merge.
2. **Параметр защищён и валидируется дважды.** Имя ветки — через `SAFE_BRANCH_RE`
   (запрет ведущего `-`) и `shq`; номер PR берётся из `findOpenPr`, где с #251 стоит
   фильтр `/^\d+$/` на входе (argument-injection закрыт до подстановки).
3. **Первый кандидат #252.** Из оставшихся мутаций эти — ближе всех к решению гейта,
   поэтому в #252 переводятся в первую очередь.

### Фетчи хореографии (`refreshRunnerWorktree`/`ensureRunnerWorktree`/`phaseDiffFiles`) — мутации не на пути автодеплоя

**Что это.** `git fetch`/`checkout --detach` обновления дерева раннера и `git fetch`
перед вычислением диффа фазы, через `shFn` (дефолт — `sh`), с `shq` на имени ветки:

```javascript
shFn(`git -C ${shq(worktreePath)} fetch origin main --quiet`); // refreshRunnerWorktree
shFn(`git -C ${shq(worktreePath)} checkout --detach origin/main --quiet`);
shFn('git fetch origin main'); // ensureRunnerWorktree
shFn(`git fetch origin main ${shq(branch)} --quiet`); // phaseDiffFiles
```

**Обоснование остаться на `sh()` (пока):** та же семья `git fetch`/`checkout`, что
переведена на argv в гейте (#193), но эти вызовы обновляют/читают дерево раннера ВНЕ
пути автодеплоя (detach уже на `origin/main`, не на голову проверяемого PR). Значения —
`origin/main` (статика) или имя ветки под `SAFE_BRANCH_RE`+`shq`. Кандидаты #252.

### Читающий `git rev-parse` головы ветки в `checksGreen` — не мутация

**Что это.** Проверка H3 (локальный ref ветки == голова PR) читает локальную голову
через `shFn` (дефолт — `sh`); комментарий у вызова ссылается на это обоснование (#194):

```javascript
localHead = shFn(`git rev-parse --verify --quiet ${shq('refs/heads/' + branch)}`);
```

**Обоснование остаться на `sh()`:** это ЧТЕНИЕ, а не мутация — argv не даёт выигрыша на
критическом пути (мутации детача/мерджа рядом уже на argv). Имя ветки квотируется `shq`
и валидируется `SAFE_BRANCH_RE` до вызова. Тот же класс, что читающие `git rev-parse`
позиции раннера в `monitor.js` (ниже).

### Вызовы `sh()` в `monitor.js`

#### Чтение progress (`issuesProgress`) — иссьюс в милестоуне

**Что это.** Функция `issuesProgress(milestone)` считает открытые/закрытые issues
в milestone для панели мониторинга. DI-параметра `shFn` у неё нет — зовёт модульный
`sh` напрямую (в отличие от соседней `openPhasePRs(config, state, shFn)`).

```javascript
const open = sh(
    `gh issue list --milestone ${shq(milestone)} --state open --json number --jq "length"`,
);
const closed = sh(
    `gh issue list --milestone ${shq(milestone)} --state closed --json number --jq "length"`,
);
```

**Обоснование остаться на `sh()`:**

1. **Только чтения для мониторинга.** gh issue list — query без side-effects. Панель — read-only.

2. **monitor.js — полностью read-only модуль.** Самостоятельный скрипт (не require ralph.js).
   Мутаций в модуле вообще нет; свежесть `ralph.log` — признак жизни раннера, монитор в
   него не пишет (инвариант 12 CLAUDE.md ralph).

3. **Параметры защищены.** Milestone — квотируется через shq().

#### Чтение git информации (`git rev-parse` позиции раннера) — панель

**Что это.** Чтение текущей ветки и HEAD для панели мониторинга:

```javascript
const head = sh('git rev-parse --short HEAD');
const branch = sh('git rev-parse --abbrev-ref HEAD');
```

**Обоснование остаться на `sh()`:**

1. **Только чтения без параметров.** Команды не берут пользовательский ввод — статические
   строки без интерпретации. Shell-injection невозможен.

2. **Мониторинг, не критичен для гейта.** Используется для вывода информации в панель.

3. **Уход на argv-вариант здесь бессмыслен.** execFileSync('git', ['rev-parse', ...])
   выглядит громоздче без выигрыша: нет переменных, которые нужно защищать.

### Итого

**Полная таблица остатка `sh()`/`shFn` (все боевые вызовы, кроме команд-чеков гейта,
которые исполняют build/lint/test проверяемого PR — они принципиально через шелл).**
Разбивка «чтение / мутация» и почему остаётся:

| Точка (`ralph.js`, если не указано)               | Ч/М         | Почему остаётся на `sh()`                                | Кандидат #252 |
| ------------------------------------------------- | ----------- | -------------------------------------------------------- | ------------- |
| `ghJson` (все читающие gh: list/view/api GET)     | чтение      | ретрай-обвязка дорога в переносе; `shq` на параметрах    | да            |
| `sweepClosedMilestones` / `closeMilestoneByTitle` | мутация     | fail-open, не на пути автодеплоя; номер из API           | да            |
| `syncProjectBoard` (`node scripts/project-sync`)  | мутация     | fail-open, косметика доски; без внешних значений         | да            |
| `removeBlockedLabel` / `addBlockedLabel`          | **мутация** | путь РЕШЕНИЯ гейта (#217), не автодеплоя; `shq`+regex    | **да (1-е)**  |
| `refreshRunnerWorktree` (`fetch`/`checkout`)      | мутация     | обновление дерева раннера вне пути автодеплоя            | да            |
| `ensureRunnerWorktree` (`git fetch origin main`)  | мутация     | подготовка worktree; `origin/main` статикой              | да            |
| `phaseDiffFiles` (`git fetch` перед diff)         | мутация     | дифф фазы для роутинга ревью; ветка под `SAFE_BRANCH_RE` | да            |
| `checksGreen` (`git rev-parse --verify` H3)       | чтение      | не мутация; `shq`+`SAFE_BRANCH_RE`                       | нет (чтение)  |
| `preflight` (`git rev-parse`, `gh auth status`)   | чтение      | статические строки, без параметров                       | нет (чтение)  |
| `monitor.js` `issuesProgress` (`gh issue list`)   | чтение      | read-only панель; `shq` на milestone                     | нет (панель)  |
| `monitor.js` `git rev-parse` позиции раннера      | чтение      | read-only панель; статические строки                     | нет (панель)  |

**Счётчики.** Прямых боевых вызовов `sh()`/`shFn` в `ralph.js` — порядка двух десятков
(плюс команды-чеки гейта, которые остаются через шелл по определению). Из них МУТАЦИЙ вне
пути автодеплоя — шесть семей (метки blocked ×2, milestone ×2, project-sync, фетчи
хореографии), остальное — чтения. Прежние формулировки «~14» и «все 7+ вызовов» были
неточны и занижали остаток — таблица выше сверена с кодом.

**Общий принцип:** argv в фазе 5 применён к мутациям **на пути к автодеплою**
(`gh pr merge`, `git fetch`/`checkout` детача PR-головы — #193). Остаток — чтения (класс
риска оплачен квотированием `shq`, не отсутствует) и мутации ВНЕ пути автодеплоя (метки,
milestone, синк доски, фетчи хореографии). Полный уход остатка на argv — #252.

**Риск остатка:** класс — строка через шелл (скованная `shq`). Не «безопасно, потому что
read-only»: инъекция в читающую `sh()` — тот же RCE-канал (инвариант 7 CLAUDE.md ralph);
сейчас он структурно закрыт квотированием, argv закрыл бы его без квотирования. Финальный
уход — #252 (после стабилизации основного цикла).
