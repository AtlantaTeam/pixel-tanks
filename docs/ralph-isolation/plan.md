# Plan: Изоляция и права автономного раннера ralph

**PRD:** `docs/ralph-isolation/prd.md`
**Дата:** 2026-07-22

Очерёдность следует разделу «Очерёдность/бюджет» PRD: дешёвое и не блокирующее — лок и
fine-grained токен — сразу; канарейка и env-санация — следом (канарейка раньше санации:
сначала измеряем, какие каналы к секретам реально открыты, потом закрываем); argv для
мутаций гейта и research по толстой границе — последними.

**Ограничения, добавленные ревью 2026-07-22.** (1) Фазы 4–5 отсюда правят те же функции
(`checksGreen`, `tryMergePhase`), что и фазы 3–5 плана наблюдаемости, — параллельно не
ведутся. (2) Часть задач фаз 1–2 требует человека (решение ЛПР, выпуск PAT в настройках
GitHub) — они помечены `needs-human`, раннер их не берёт.

## Фазы реализации

### Фаза 1: Лок от двойного запуска

**Цель:** два раннера на одном состоянии физически не стартуют.
**Затрагивает:** ralph-раннер (`ralph.js`), файловое состояние в `.claude/ralph/`

**Задачи:**

- [ ] Подтвердить у ЛПР гранулярность лока (рекомендация PRD: один лок на
      репозиторий-машину в `.claude/ralph/` дерева человека, `.gitignore`; playground и
      prod на одной машине — ОДИН лок) и записать решение в plan/research.
- [ ] Файл-лок с pid; живость — `kill(pid, 0)` + сверка `/proc/<pid>/cmdline` на
      `ralph.js` по образцу `adoptMonitor`/`isMonitorProcess`.
- [ ] Живой лок → отказ fail-closed (не подхват и не ожидание), сообщение называет pid и
      путь лок-файла. Осиротевший лок (процесс мёртв или cmdline чужой) → снимается
      автоматически, событие в лог.
- [ ] Взятие лока — до любых побочек: state, лог живого прогона, git, монитор не тронуты.
- [ ] Тесты (DI, `RALPH_NO_SIDE_EFFECTS=1`, `guardSideEffect`): отказ при живом локе без
      побочек, автоснятие сироты после `kill -9`, негативный тест на переиспользованный
      pid чужого процесса, битый/нечитаемый лок-файл → стоп с внятным сообщением.

**Когда готова:** второй запуск при живом первом падает до любых побочек с сообщением о
pid и пути лока; запуск после `kill -9` проходит без ручной чистки.

### Фаза 2: Мердж-токен минимальных прав

**Цель:** утечка токена петли ограничена одним репозиторием.
**Затрагивает:** секреты VDS (`/root/ralph.env`), настройки GitHub

**Задачи:**

- [ ] Research: матрица «шаг петли → нужные права» (кодер-сессия — issues/PR/push, гейт —
      чтение PR + мердж, ревью — комменты); зафиксировать, насколько достижим идеал
      «токен мерджа доступен только шагу мерджа» без толстой границы. Учесть шаги,
      которые добавляет план наблюдаемости: `gh run` в пост-мердж проверке (право
      Actions: read) и чтение комментариев PR в метрике находок — иначе первый же
      пост-мердж чек упадёт ночью с «не хватило прав».
- [ ] Выпустить fine-grained PAT только на `AtlantaTeam/pixel-tanks` с минимальным
      набором прав по матрице; заменить текущий токен в env раннера.
- [ ] Ручная проверка: операция этим токеном в другом репозитории отклоняется GitHub —
      результат зафиксировать в research.
- [ ] Приёмочный прогон полного цикла фазы (кодер → PR → ревью → правки → гейт → мердж)
      на playground: отказов «не хватило прав» нет.

**Когда готова:** петля мерджит fine-grained токеном, ограниченным одним репозиторием;
полный цикл фазы на нём проходит без отказов по правам.

### Фаза 3: Канареечная проверка границы

**Цель:** «изолировано» становится проверяемым фактом, а не верой.
**Затрагивает:** тесты/гейт-чеки

**Задачи:**

- [ ] Канареечный тест — отдельный скрипт/тест, запускаемый вручную (в гейт он встаёт
      только в фазе 4, сразу красным чеком: до санации это был бы вечно-зелёный чек,
      который находит секреты и молчит). Пытается достать
      `GH_TOKEN`, `CLAUDE_CODE_OAUTH_TOKEN`, `TG_BOT_TOKEN` из `process.env` и по
      известным файловым путям токенов (в т.ч. `~/.config/gh/hosts.yml`).
- [ ] Фикстура npm-зависимости с lifecycle-скриптом (postinstall) — тот же канареечный
      подход для `npm ci`.
- [ ] Установить фактически, откуда `gh` на VDS берёт токен (env или `hosts.yml`) —
      до выбора границы; записать в research.
- [ ] Зафиксировать baseline «что видно сегодня» — до санации канарейка ожидаемо
      находит секреты; это измерение, а не красный гейт.

**Когда готова:** канарейка запускается вручную и даёт воспроизводимый отчёт, какие
каналы к секретам открыты; фактический источник токена `gh` на VDS задокументирован.
В гейте на этой фазе канарейки нет.

### Фаза 4: Env-санация чеков гейта

**Цель:** чеки гейта и `npm ci` исполняются без секретов в окружении.
**Затрагивает:** ralph-раннер (`checksGreen`, `installFn` в `ensureRunnerWorktree` /
`syncDepsIfLockChanged`)

**Задачи:**

- [ ] Allowlist нужных переменных (не blocklist — новый секрет не должен молча
      наследоваться), файл в репозитории: изменения видны в диффе и попадают под
      эскалацию ревью (`.claude/ralph/**` уже в `review.escalateOnPaths`).
- [ ] Санировать окружение в точках перехвата: цикл чеков в `checksGreen` и `installFn`.
- [ ] Канарейка из фазы 3 переводится в обязательный красный чек: находка секрета —
      ненулевой exit-код с сообщением, отличающим «секрет найден» от «переменная не в
      allowlist».
- [ ] Проверить, что толстые prod-чеки не сломаны: e2e сохраняет кэш браузеров
      Playwright и поднимает свой dev-сервер.
- [ ] Честно записать остаточный риск: (а) — не песочница, файловый канал от того же
      пользователя ОС остаётся открытым до варианта (б)/(в).

**Когда готова:** канарейка и postinstall-фикстура не находят секретов; полный гейт
(e2e, coverage, security) зелёный на эталонном заведомо-зелёном коммите после включения
санации.

### Фаза 5: argv для мутаций гейта

**Цель:** мутации на пути к автодеплою прода не идут значениями сквозь шелл.
**Затрагивает:** ralph-раннер (`checksGreen`, `tryMergePhase`)

**Задачи:**

- [ ] Перевести `gh pr merge` и `git fetch`/`checkout` в `checksGreen`/`tryMergePhase` на
      `execFileSync` с argv.
- [ ] Для каждого вызова, который решено оставить на `shq()`, записать обоснование в
      research (полный рефактор всех ~14 мест `sh()` — отдельное решение, не эта фаза).
- [ ] Тесты на изменённые вызовы: значения с пробелами и спецсимволами не интерпретируются
      шеллом; побочки — через DI.

**Когда готова:** мутации гейта вызываются через argv либо имеют явно записанное
обоснование остаться на `shq()`.

### Фаза 6: Открытые вопросы в бэклог

**Цель:** отложенные решения видны на доске, а не растворяются в PRD.
**Затрагивает:** бэклог GitHub

**Задачи:**

- [ ] Issue: выбор толстой границы — отдельный пользователь ОС (б) vs контейнер/namespace
      (в), с обкаткой на playground (не дублировать Docker/E2B-issue из `ralph-prod-mode`).
- [ ] Issue: дифф PR в промпте ревью — варианты снижения риска (файл-вложение,
      отдельная сессия-пересказчик); наращивать инструкции промпта запрещено брифом.
- [ ] Issue: гранулярность лока — только если ЛПР не подтвердил рекомендацию в фазе 1.
- [ ] Все — с метками `backlog`, `area:devops`, `complexity:*` и на доске проекта.

**Когда готова:** на каждый открытый вопрос PRD заведён issue со ссылкой на раздел PRD,
с метками и карточкой на доске.
