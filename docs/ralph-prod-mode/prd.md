# PRD: Прод-режим ralph

**Дата**: 2026-07-19
**Статус**: Draft

## Цель

Превратить автономный loop ralph (обкатан на game-next, фазы 3-9) в конвейер,
которому можно доверить работу над **боевым приватным репозиторием**: раннер
изолирован от рабочего дерева человека, гейт толще (не только зелёная сборка, но
e2e/coverage/security), а точка вмешательства человека сдвинута с **мерджа** на
**релиз/деплой**. Значительная часть надёжности уже реализована в `ralph.js` (см.
«Что уже есть»); этот PRD описывает только дельту прод-режима, не переписывая её.

## Что уже есть (НЕ переделываем — фиксируем как факт)

Реализовано в `.claude/ralph/ralph.js` и обкатано на фазах 3-9:

- **Внешний loop**, 1 issue = 1 чистая сессия; state адресует фазу по имени milestone.
- **Детерминированный гейт мерджа**: `blocked`-label + сверка локального HEAD с головой
  PR (fetch) + 5 зелёных чеков `build / lint / lint:fsd / typecheck / test`. Мердж —
  squash, только по зелёному.
- **LLM-ревью фазы** отдельной моделью (opus, эскалация на fable по `complexity:expert`),
  покрывает архитектуру/безопасность/perf/PRD/нейминг/тесты; блокеры → label `blocked`.
- **Self-heal**: красный чек → чини-сессия с текстом ошибки (до `gateHealAttempts`, деф. 2);
  `blocked` от ревью → авторазбор (до `blockedHealAttempts`, деф. 3); обе идут через
  повторное ревью. **merge-retry** с проверкой `phaseMerged()` (не задвоить).
- **Circuit breaker**: `maxIterations` (фаза), `maxTurns` (сессия), `maxNoProgress`
  (итераций без коммита/закрытого issue, деф. 3). Счётчики durable в `ralph.state.json`.
- **API-лимит auto-wait** (идея frankbria): маркер лимита → парсинг «resets 1:20pm» →
  сон до сброса окна + повтор (до `apiLimitMaxWaits`, деф. 3). Недельный лимит подписки
  раннер переживает сам.
- **Роутинг моделей** по label `complexity:{low|medium|high|expert}` → haiku/sonnet/opus/fable.
- **Инъекция-защита (C3, код-слой)**: `authorAllowlist` (fail-closed при пустом),
  промпты ревью/правок/blocked велят игнорировать чужие комментарии.
- **Авто-закрытие milestone** после мерджа + свип на старте.
- **`monitor.js`** — read-only панель прогресса (фаза, issues, PR, хвост лога), запускается
  **отдельно вручную**.

**Открытым остаётся операционный слой C3**, прямо описанный в комментарии `ralph.js`:
публичный репо + `bypassPermissions` = поверхность инъекции; private-репо и запуск в
песочнице/VM — «за человеком, раннер видимость репо не меняет». Это и есть входной билет
прод-режима.

## Пользовательские сценарии

- Человек запускает `ralph.js --profile prod` → раннер работает в **отдельном git
  worktree**, не трогая дерево человека; при старте сам поднимает `monitor.js`, при
  выходе глушит его.
- Loop берёт issue → реализует → **толстый** детерминированный гейт (5 текущих чеков +
  e2e + coverage-порог + security-скан) → зелёно → squash-merge сам; `blocked` → **сразу
  человеку** (в prod `blockedHealAttempts=0`, авторазбора нет).
- Все issues фазы смерджены → loop **останавливается на границе релиза**, шлёт пуш
  «фаза готова к релизу»; деплой человек жмёт сам (в playground мердж = финал, как сейчас).
- Человек правит/коммитит в своём дереве во время работы loop — `ensureClean` раннера
  не падает, потому что деревья разведены (сегодня общее дерево это ломает —
  см. `feedback-ralph-shared-worktree`).
- Ключевые события (релиз-готовность, `blocked`, breaker, rate-limit) приходят пушем в
  **Telegram-бот**, а не только в консоль `monitor.js`.
- Прод-режим исполняется на **удалённом Linux-сервере вне РФ** (Frankfurt): домашний комп
  не нагружен, прокси к Anthropic не нужен (не-РФ IP), ralph крутится под systemd,
  переживая разрыв SSH; человек смотрит код через VS Code Remote-SSH.

## В скоупе (MVP)

0. **Портирование `ralph.js`/`monitor.js` на Linux (входной билет — до всего остального).**
   Раннер писался и гонялся ТОЛЬКО на Windows. На Linux `spawnSync({shell:true})` зовёт
   `/bin/sh`, а не `cmd.exe` → guard `%`/`"` (заточен под cmd.exe-раскрытие) переосмыслить;
   проверить кавычки/экранирование промптов, пути (сепараторы), detached-spawn монитора,
   `git`/`gh`-вызовы. Без зелёного прогона всех веток кода на Linux прод-режим не поедет.

1. **Изоляция раннера через git worktree.** Раннер работает в выделенном worktree
   (напр. `../pixel-tanks-ralph/`). Требует пересмотра хореографии веток: сейчас
   `checksGreen`/`checkoutMainQuiet`/`tryMergePhase` делают `git checkout` в общем дереве —
   в модели worktree они оперируют деревом раннера, не человека. Закрывает shared-worktree
   `ensureClean`-конфликт и операционный слой C3 частично (та же ОС/сеть/ключи).

2. **Профили конфига `playground` / `prod`** в `ralph.config.json`.
    - Сейчас конфиг плоский — вводим именованные профили. Наследуют общие поля
      (`modelRouting`, `review`, `phases`), переопределяют лишь дельту.
    - `playground` = текущее поведение (`blockedHealAttempts=3`, гейт 5 чеков, публичный
      репо, мердж = финал).
    - `prod` = `blockedHealAttempts=0`, толстый гейт, приватный репо, release-стоп,
      пуш-уведомления. Значения-крутилки (`blockedHealAttempts` и т.п.) уже параметризованы —
      профиль лишь их бандлит, механики не трогает.

3. **Толстый детерминированный гейт (prod).** Расширить `GATE_CHECKS` сверх нынешних пяти:
    - **e2e** (Playwright) — критический путь;
    - **coverage-порог** — конкретное число в research (сейчас: «покрытие не падает» vs жёсткий порог — решить);
    - **детерминированный security-скан** (`npm audit` / semgrep) — это ДОБАВКА к уже
      существующему LLM-ревью безопасности, не замена;
    - staging-деплой + smoke — **в бэклог до появления реального деплоя** (не MVP).
    - Гейт остаётся детерминированным: любой красный чек блокирует мердж.

4. **Сдвиг точки человека merge→release.** Сегодня мердж в main = конец конвейера
   (deploy-стадии нет). В prod: loop мерджит по зелёному, но по закрытию всех issues фазы
   **останавливается перед деплоем** и шлёт пуш «готово к релизу»; деплой — человек.

5. **`monitor.js` авто-старт/стоп.** `ralph.js` при старте спавнит `monitor.js`
   детачнутым child-процессом, при выходе глушит. Убрать двойной ручной запуск.

6. **Пуш-уведомления в Telegram (prod), односторонние.** Сегодня события видны только в
   консоли. Добавить пуш-нотифаер на 4 события: релиз-готовность, `blocked` отдан человеку,
   breaker открылся, упёрлись в rate-limit. Канал — **Telegram-бот** (`@BotFather` токен →
   `sendMessage`), только бот→человек. События раннер знает из первых рук (уже логирует их) —
   откуда именно эмитить пуш (ralph.js в точках событий vs monitor.js по логу) решает plan.
   Секреты (`TG_BOT_TOKEN`, `chat_id`) — в env `600`, не в конфиг.

## Не в скоупе (бэклог — наша новизна, референса нет → риск выше)

- **Docker/E2B песочница** — полная изоляция ОС/сети/секретов поверх worktree.
- **Параллелизм независимых issues** в отдельных worktrees + координация мерджей
  (у frankbria очередь строго последовательная — целиком наша надстройка).
- **Durable-семантика уровня Temporal** — чекпоинты/replay движком (сейчас state в JSON,
  ретраи/идемпотентность дописаны руками; у frankbria — тоже лишь файловые блобы).
- **Событийные триггеры** — loop + реакции (CI упал → чини, issue → триаж, security-алерт → ревью).
- **Фанаут-эскалация** — noProgress-breaker вместо стопа: 2-3 агента разными подходами + судья.
- **Мульти-критик ревью** — независимые критики параллельно (security + perf + архитектура);
  субагенты в `.claude/agents/` есть, loop их пока не зовёт.
- **staging-деплой + smoke в гейте** — до появления реального деплой-таргета.
- **Двусторонний Telegram-бот** (управление раннером) — вместе с event-триггерами. Лестница:
  сначала read-only команды (`/status`, `/log`, `/screenshot` — отражают monitor), потом
  мутирующие (`/pause`, `/merge`, `/approve`) с авторизацией по `chat_id`. На MVP — только пуши.
- **Docker/E2B-песочница поверх сервера** — если Linux-сервер+worktree окажется мало по C3.

**Отвергнуто осознанно (не делаем никогда):**

- **EXIT_SIGNAL / completion-эвристики** (frankbria) — наш выход по закрытию GitHub issues
  детерминированнее.
- **Долгоживущий агент с самокомпакцией** — дрейф; держим «1 issue = 1 чистая сессия».

## Технические ограничения

- Раннер — Node.js (`ralph.js`, `monitor.js`), синхронная execSync-хореография; конфиг —
  `ralph.config.json`, состояние — `ralph.state.json`, правила агента — `ralph.md`.
- **Целевая среда прод-режима — удалённый Linux-сервер вне РФ** (Ubuntu LTS, Frankfurt),
  поднимается provisioning-скриптом (Node, gh, claude CLI, Playwright+chromium, swap, ufw
  SSH-only, systemd-юнит для ralph). Скрипт — infra-деливери рядом с PRD, вне gate-логики.
  Следствия для кода: **прокси к Anthropic не нужен** (не-РФ IP снимает Cloudflare-403,
  прежний костыль `NODE_USE_ENV_PROXY` отпадает); shell на Linux — `/bin/sh`, поэтому
  win32-обоснование prompt-guard (`%`/`"` под cmd.exe) устарело и переосмысливается в п.0.
- **Целевой прод-репо — будущий приватный проект, НЕ pixel-tanks** (он — публичный учебный
  полигон = `playground`). Prod-профиль проектируем переносимым, обкатываем на playground.
- **Формат работы над PRD и конвейером — только человек (Дима) вручную в консоли, НЕ через
  loop.** Проектирование конвейера не доверяем самому конвейеру.
- Секреты (`ANTHROPIC`, `GH`, `TG_BOT_TOKEN`, `chat_id`) — в env-файле `600` на сервере,
  читать node-скриптом, не коммитить.
- **merge→release при отсутствии реального деплой-таргета** (pixel-tanks не деплоится): в
  prod «release-стоп» реализуется как «стоп + пуш вместо продолжения», деплой-шаг — плейсхолдер
  до появления боевого проекта. Механизм проверяем, деплой пока no-op.
- Модель-роутинг и review-эскалация — сохранить, профили их не ломают.

## Критерии готовности (MVP)

- [ ] `ralph.js` и `monitor.js` проходят полный прогон на Linux (Ubuntu LTS): все ветки кода
      (промпты с guard, пути, detached-spawn монитора, git/gh) работают под `/bin/sh` без регресса.
- [ ] `ralph.js --profile prod` работает в отдельном git worktree; правки/коммиты человека
      в основном дереве не роняют `ensureClean` раннера.
- [ ] Хореография веток (`checksGreen`/`checkoutMainQuiet`/`tryMergePhase`) корректна в
      worktree-модели: чеки гоняются на коде ветки раннера, мердж по-прежнему по зелёному.
- [ ] В `ralph.config.json` есть профили `playground` и `prod`; `playground` полностью
      воспроизводит текущее поведение (регресса нет), `prod` меняет только заявленную дельту.
- [ ] В `prod` `blocked`-issue сразу человеку (0 попыток), в `playground` — до 3 (как сейчас).
- [ ] Толстый гейт prod прогоняет e2e + coverage-порог + security-скан вдобавок к 5 текущим
      чекам; красный любой из них блокирует мердж (детерминированно).
- [ ] В `prod` loop мерджит по зелёному, но по закрытию всех issues фазы **не деплоит** —
      останавливается и шлёт пуш «готово к релизу». В `playground` мердж остаётся финалом.
- [ ] `ralph.js` при старте сам запускает `monitor.js`, при выходе глушит; отдельный ручной
      запуск монитора не требуется.
- [ ] Пуш-уведомления в **Telegram** приходят на 4 события (релиз-готовность, blocked,
      breaker, rate-limit) в профиле prod; токен/`chat_id` — из env, не из конфига.
- [ ] Все пункты «не в скоупе» заведены отдельными issues в бэклог (не потеряны).
