# Plan: Прод-режим ralph

**PRD:** [prd.md](./prd.md)
**Дата:** 2026-07-19

## Фазы реализации

### Фаза 1: Раннер на Timeweb-сервере (РФ) + provisioning + туннель (Tracer Bullet)

**Цель:** `ralph.js`/`monitor.js` целиком работают на Ubuntu (Timeweb VDS, Москва) в текущем
(playground) поведении. Обкатка provisioning — на **эфемерном Timeweb-VDS через MCP** (создал →
провижн → снёс, почасовка; VirtualBox не нужна). Тонкий сквозной срез: окружение поднято, туннель
к Anthropic жив, раннер прогоняет итерацию.
**Затрагивает:** runner (ralph.js, monitor.js), infra (provisioning-скрипт, туннель)
**Задачи:**

- [ ] Аудит кросс-платформенных точек: guard `%`/`"` (заточен под cmd.exe), кавычки/экранирование
      промптов, разделители путей, detached-spawn монитора, вызовы `git`/`gh` под `/bin/sh`.
- [ ] Адаптировать `spawnSync({shell:true})`-хореографию под `/bin/sh` без регресса поведения.
- [x] **Provisioning-скрипт** (`.claude/ralph/provision/`): Node 24, gh, Claude CLI,
      Playwright+chromium, Shadowsocks-туннель (ss-local→privoxy), ufw SSH-only, env-файл `600`
      (CLAUDE*CODE_OAUTH_TOKEN/GH/SS/HTTPS_PROXY), clone+`npm ci`. Идемпотентен, гоняется на
      эфемерном Timeweb-VDS через MCP. *(готово 2026-07-20)\_
- [x] **Туннель к Anthropic** (сервер в РФ): claude через `HTTPS_PROXY=http://127.0.0.1:8118` →
      privoxy → ss-local → Outline вне РФ; health-check egress == прокси-IP. _(проверено вживую)_
- [ ] Unit-тесты на изменённые чистые функции (guard/путь/парсинг) + прогон существующих тестов на Linux.
- [ ] Прогон `--dry-run` и playground-итерации на Timeweb-VDS (живые claude-вызовы идут через туннель).

**Когда готова:** provisioning-скрипт из чистого Ubuntu (эфемерный Timeweb-VDS) разворачивает
окружение + туннель, и `node ralph.js --dry-run` + playground-итерация проходят без регресса.

### Фаза 2: Профили playground/prod + авто-старт монитора

**Цель:** `--profile` выбирает бандл настроек; профиль `prod` существует; `ralph.js` сам
поднимает `monitor.js`.
**Затрагивает:** runner (config-слой, ralph.js)
**Задачи:**

- [ ] Схема профилей в `ralph.config.json`: общие поля (`modelRouting`/`review`/`phases`)
      наследуются, профиль переопределяет только дельту.
- [ ] Флаг `--profile <name>` + резолв; дефолт = playground.
- [ ] `prod` бандлит крутилки (`blockedHealAttempts=0` и пр.); `playground` = текущее поведение.
- [ ] `ralph.js` спавнит `monitor.js` детачнутым child при старте, глушит при выходе.
- [ ] Тесты: merge/резолв профилей; `playground` — регресс-тест (поведение не изменилось);
      `prod` — `blockedHealAttempts=0`.

**Когда готова:** `--profile playground` не меняет поведения (регресса нет); `--profile prod`
переключает blocked на 0; монитор запускается автоматически, отдельный ручной запуск не нужен.

### Фаза 3: Изоляция раннера в git worktree

**Цель:** loop работает в выделенном worktree, человек правит основное дерево не мешая.
**Затрагивает:** runner (ветковая хореография)
**Задачи:**

- [ ] Поднять/использовать выделенный worktree раннера (напр. `../pixel-tanks-ralph`).
- [ ] Переработать `checksGreen`/`checkoutMainQuiet`/`tryMergePhase`: `git checkout`
      оперирует деревом раннера, не человека.
- [ ] `ensureClean` проверяет дерево раннера, изолированно от основного.
- [ ] Тесты на ветковые хелперы (мок git); сценарий «правки человека в main-дереве не роняют loop».

**Когда готова:** человек коммитит в основном дереве во время работы loop — `ensureClean`
раннера не падает; мердж по-прежнему только по зелёному гейту.

### Фаза 4: Толстый детерминированный гейт (prod)

**Цель:** prod-гейт добавляет e2e + coverage-порог + security-скан сверх 5 текущих чеков.
**Затрагивает:** runner (GATE_CHECKS), тесты e2e
**Задачи:**

- [ ] Состав `GATE_CHECKS` зависит от профиля: playground = 5, prod = 5 + e2e + coverage + security.
- [ ] e2e-чек (Playwright, критический путь игры) — headless на сервере.
- [ ] coverage-порог (число из research; `--coverage` с threshold) — красный при падении ниже.
- [ ] Детерминированный security-скан (`npm audit`/semgrep) — добавка к LLM-ревью, не замена.
- [ ] Тесты: выбор набора чеков по профилю; красный любого нового чека блокирует мердж.

**Когда готова:** в prod любой красный из e2e/coverage/security блокирует мердж; playground —
прежние 5 чеков без изменений.

### Фаза 5: Telegram-пуши + сдвиг merge→release

**Цель:** события уходят в Telegram; в prod точка человека — релиз, не мердж.
**Затрагивает:** runner (нотифаер, release-стоп)
**Задачи:**

- [ ] Модуль TG-нотифаера: `sendMessage`, токен/`chat_id` из env `600`, только бот→человек.
- [ ] Эмит 4 событий: релиз-готовность, `blocked`→человеку, breaker открылся, rate-limit.
- [ ] prod release-стоп: мердж по зелёному, но по закрытию всех issues фазы — стоп + пуш
      «готово к релизу» (деплой — плейсхолдер до боевого таргета). playground: мердж = финал.
- [ ] Тесты: форматирование сообщений + триггеры событий (мок TG); prod останавливается на релизе.

**Когда готова:** 4 события приходят в Telegram (prod); prod встаёт на границе релиза с пушем;
playground сохраняет мердж=финал.

### Фаза 6: Бэклог в issues (закрытие PRD-критерия)

**Цель:** все пункты «не в скоупе» заведены как отслеживаемые issues, ничто не потеряно.
**Затрагивает:** GitHub (issues, доска)
**Задачи:**

- [ ] Завести issues на каждый бэклог-вектор: Docker/E2B, параллелизм, durable-replay,
      event-триггеры, фанаут-эскалация, мульти-критик, staging-smoke, двусторонний TG-бот,
      **S3-бэкап транскриптов (Timeweb S3), golden-образ VDS (create_image), мониторинг железа
      (get_server_statistics)**.
- [ ] Пометить label `backlog`, добавить на доску.

**Когда готова:** каждый бэклог-пункт PRD имеет tracking-issue на доске проекта.
