# Работа на VDS: сессии, tmux и что выживает при разрыве

Шпаргалка для работы на `ralph-prod` (VS Code Remote-SSH). Для тех, кто всю
жизнь работал на локалке.

## Главная идея

Окно VS Code на ноуте — только «пульт». Всё настоящее (файлы, терминалы,
claude, ralph) живёт **на VDS**. Вопрос лишь в том, что умирает, когда пульт
отключился.

## Что умирает при разрыве SSH, а что нет

У процесса, запущенного из терминала, есть «поводок» — управляющий терминал
(колонка TTY в `ps`). Разорвал SSH → терминал исчез → всем на поводке прилетает
SIGHUP → умирают. Кто без поводка (tmux, systemd-службы) — живёт дальше.

| Что                                               | Разрыв SSH                                                               |
| ------------------------------------------------- | ------------------------------------------------------------------------ |
| Терминал VS Code и запущенное в нём               | 💀 умирает (vscode-server иногда восстанавливает — это кэш, не гарантия) |
| `claude`, запущенный в терминале напрямую         | 💀 умирает                                                               |
| `claude` внутри tmux                              | ✅ живёт                                                                 |
| ralph, запущенный ЧЕРЕЗ claude фоновым Bash-тулом | 💀 умирает — см. предостережение ниже                                    |
| ralph, запущенный напрямую в tmux                 | ✅ живёт                                                                 |
| ss-local / privoxy (systemd)                      | ✅ живут (`Restart=always`)                                              |

**Проверено на практике боевым прогоном 2026-07-20, а не в теории.** Первая
попытка: ralph запущен фоновым Bash-тулом claude (`run_in_background: true`).
Процесс выглядел «отвязанным от терминала» (свой PGID/SID, TTY `?`) — отсюда
была гипотеза «переживёт разрыв, как обычный демон». Гипотеза оказалась
НЕВЕРНОЙ: когда прошлая claude-сессия закрылась, ralph умер вместе с ней ровно
на шаге гейта мерджа (лог оборвался посреди прогона чеков, PR остался
несмерджен, никакого OOM/kill в системных логах не найдено — значит, дело
не в системе, а в самом Claude Code). Похоже, CLI трекает и подчищает свои
фоновые Bash-задачи вместе со своим процессом — это его собственный жизненный
цикл, SIGHUP тут ни при чём. Вторая попытка: тот же ralph запущен НАПРЯМУЮ в
tmux-сессии (`tmux send-keys`, минуя claude-инструмент) — вот это настоящая
независимость: гейт дошёл до конца и смерджил PR без сучка.

**Вывод: любой прогон ralph длиннее пары минут — только через tmux напрямую,
никогда через фоновый Bash claude.** Фоновый Bash claude годится для команд на
секунды-минуты в рамках одной сессии, не для AFK-циклов на часы.

## tmux: долгоиграющие сессии

tmux — терминальная сессия, живущая на сервере сама по себе. Ты к ней
подключаешься (attach) и отключаешься (detach), как к вечно идущему фильму.

### Ежедневный сценарий

```bash
w                  # alias: войти в сессию work (создаст, если нет)
claude --continue  # продолжить последний разговор (или просто claude)
# ... работаешь ...
# Ctrl+B, потом D — отцепился; ноут можно закрывать
# ...завтра...
w                  # тот же разговор, та же сессия, всё на месте
```

### Запуск ralph — только так

```bash
w                                             # войти в tmux
IS_SANDBOX=1 node .claude/ralph/ralph.js      # напрямую в панели tmux, БЕЗ claude
# Ctrl+B, D — отцепиться; ralph продолжит работать
```

Или из другой сессии, не заходя в панель глазами:

```bash
tmux send-keys -t work 'IS_SANDBOX=1 node .claude/ralph/ralph.js' Enter
tmux capture-pane -t work -p | tail -20   # подсмотреть вывод, не подключаясь
```

### Шпаргалка tmux (префикс — Ctrl+B, потом клавиша)

| Клавиши                     | Действие                                     |
| --------------------------- | -------------------------------------------- |
| `Ctrl+B D`                  | detach — выйти, оставив сессию жить          |
| `Ctrl+B %` / `"`            | разделить панель вертикально / горизонтально |
| `Ctrl+B ←↑↓→`               | переключение между панелями                  |
| `Ctrl+B C`                  | новое окно; `Ctrl+B 1..9` — переключение     |
| `Ctrl+B [`                  | скролл-режим (выход — `q`); мышью тоже можно |
| `tmux ls`                   | список сессий снаружи                        |
| `tmux kill-session -t work` | убить сессию совсем                          |

Конфиг: `/root/.tmux.conf` (мышь, история 100k строк, статус-бар).

## claude: как продолжить разговор после смерти сессии

История разговоров хранится на диске (`/root/.claude/`), поэтому:

- `claude --continue` — продолжить ПОСЛЕДНИЙ разговор этого проекта;
- `claude --resume` — выбрать из списка прошлых разговоров.

Умерла сессия — ничего не потеряно: заходишь в tmux, `claude --continue`,
контекст подхватывается (плюс SessionStart-хук подгружает HANDOFF.md).
Нюанс: фоновые задачи прежней сессии (мониторинг-лупы) к новой не
переподцепляются — статус ralph смотреть руками: `tail .claude/ralph/ralph.log`
или `tmux capture-pane -t work -p`, если ralph крутится в tmux.

Не держи два живых claude на одном разговоре одновременно: старую сессию
закрой (`/exit` или Ctrl+C) до `--continue` в tmux.

## ralph на VDS

**Монитор запускать руками больше не нужно (#74).** Раннер сам поднимает `monitor.js`
детачнутым процессом после preflight и глушит его при выходе — включая Ctrl+C и `kill`.
Панель уходит в `.claude/ralph/monitor.out` (у детачнутого процесса нет терминала):

```bash
tail -f .claude/ralph/monitor.out      # панель прогресса, из любого окна
```

Файл перезаписывается при каждом старте, pid лежит в `monitor.pid`; оба вне git.
При `--dry-run` монитор не поднимается. Если от прошлого (убитого `kill -9` / OOM)
прогона остался живой монитор — раннер по `monitor.pid` подхватит его вместо второго
и заглушит при своём выходе; перед kill сверяется `/proc/<pid>/cmdline`, чтобы не
задеть чужой процесс с переиспользованным pid. Монитор, запущенный руками, pid-файл
не пишет — раннер его не увидит и заведёт свой (двойные gh-запросы), поэтому руками
монитор больше не запускать.

Штатный способ автономного (не завязанного на человека) перезапуска ralph
на VDS — tmux-сессия (монитор раннер поднимет сам), или systemd-юнит БЕЗ
`Restart=always` (fail-closed стопы ralph осознанные — красный гейт, blocked,
мёртвый туннель; слепой авто-перезапуск бился бы об ту же стену и жёг API-лимит)
— issue #103.

Проверить, жив ли ralph, после переподключения:

```bash
pgrep -af 'ralph.js'                   # процесс
tail -f .claude/ralph/ralph.log        # живой лог (Ctrl+C — выйти)
tail -f .claude/ralph/monitor.out      # панель, поднятая самим раннером
node .claude/ralph/monitor.js --once   # разовый снимок панели (если раннер не запущен)
```
