# Plan: Наблюдаемость автономного раннера ralph

**PRD:** `docs/ralph-reliability/prd.md`
**Дата:** 2026-07-22

Очерёдность фаз следует ответу на вопрос 4 брифа: вектор 1 первым (транспорт готов —
фаза 5 prod-mode смерджена), затем защита тестов (чеки в существующий гейт), затем
пост-мердж проверка, затем метрика. Каждая фаза самодостаточна: после любой можно
остановиться, петля остаётся рабочей.

**Очерёдность относительно парной фичи (ревью 2026-07-22).** Фазы 4–5 изоляции
(`docs/ralph-isolation/plan.md`) правят те же функции — `checksGreen`, `tryMergePhase` —
и параллельно с фазами 3–5 этого плана не ведутся. Allowlist env-санации (изоляция,
фаза 4) обязан включить переменные, нужные новым чекам фаз 3–4 отсюда (репортёр vitest),
а её приёмка «полный гейт зелёный» гоняется уже с этими чеками.

## Фазы реализации

### Фаза 1: Deadman — тишина как алерт

**Цель:** зависший или мёртвый прогон становится пушем человеку, а не тишиной до утра.
**Затрагивает:** ralph-раннер (`monitor.js`, нотифаер), конфиг профилей

**Задачи:**

- [ ] Зафиксировать признак жизни (время последней записи в `ralph.log` worktree) и
      пороги тишины по последнему событию лога (после «🔄 итерация» — `claudeTimeoutMs` + запас; после «🚦 Гейт» — таймаут самого долгого чека; иначе — короткий дефолт);
      числа взять замером реальных фаз 4–5 и вынести в `ralph.config.json`.
- [ ] Реализовать детект тишины в `monitor.js` (он detached и переживает смерть
      раннера) — чтение лога снаружи процесса сессии, без heartbeat-файла.
- [ ] Отправлять пуш через `pushEvent()` нотифаера фазы 5; цикл раннера при этом НЕ
      останавливать (ложный ночной стоп дороже лишнего пуша).
- [ ] Тесты: срабатывание по каждому порогу, отсутствие срабатывания на легитимно
      долгой кодер-сессии, дедупликация повторных пушей об одной и той же тишине.
      Побочки — через DI, `RALPH_NO_SIDE_EFFECTS=1` и `guardSideEffect` обязательны.

**Когда готова:** `kill -STOP` процесса сессии → пуш о тишине не позднее порога шага;
`kill -9` раннера → пуш приходит без участия раннера; полный живой прогон фазы не даёт
ни одного ложного deadman-пуша.

### Фаза 2: Взаимный контроль раннер↔монитор

**Цель:** сторож не умирает молча — мёртвого монитора переподнимает раннер.
**Затрагивает:** ralph-раннер (`ralph.js`, `monitor.js`)

**Задачи:**

- [ ] Проверять живость монитора на каждой итерации `runLoop()` (`monitorAlive` +
      `isMonitorProcess`), а не только на старте в `main()`.
- [ ] Переподнимать мёртвый монитор с обновлением pid-файла; событие — в лог.
- [ ] Тесты: монитор убит между итерациями → к следующей итерации поднят новый, pid-файл
      обновлён; чужой процесс с переиспользованным pid не считается живым монитором.

**Когда готова:** симуляция «убить монитор посреди прогона» приводит к поднятому новому
монитору к следующей итерации, pid-файл обновлён.

### Фаза 3: Храповик числа тестов

**Цель:** гейт краснеет, когда тестов стало меньше без осознанного решения.
**Затрагивает:** гейт-чеки (`scripts/`), эталон в репозитории

**Задачи:**

- [ ] Взять число прогнанных unit-тестов из детерминированного репортёра vitest
      (json/summary), не из парсинга человекочитаемого stdout.
- [ ] Файл-эталон в репозитории по образцу `scripts/security-audit.baseline.json`:
      число + `reason` для осознанного снижения.
- [ ] Гейт-чек в `BASE_GATE_CHECKS`: падение ниже эталона → ненулевой exit-код с
      сообщением «каких тестов не досчитались»; рост → зелёный без правки эталона.
- [ ] Fail-closed: нечитаемый или неожиданный формат отчёта — красный, не «пропустим».
- [ ] Тесты, включая негативные: снижение без обновления эталона, снижение с
      обновлением и reason, битый отчёт репортёра.

**Когда готова:** PR, уменьшающий число unit-тестов без обновления эталона, красит
гейт; PR с ростом тестов и PR с легальным удалением + обновлённым эталоном — зелёные.

### Фаза 4: Детект `.only` / `.skip` в unit-гейте

**Цель:** ослабление проверок через `.only`/`.skip` не проезжает в main.
**Затрагивает:** гейт-чеки (`scripts/`), эталон

**Задачи:**

- [ ] Записать выбранный режим по `.skip` до реализации (рекомендация PRD — вариант (а),
      красный на любой новый skip; в `src/` сейчас ни одного skip — чистая точка отсчёта).
- [ ] Чек: `it.only`/`describe.only` в unit-тестах → красный гейт (аналог `forbidOnly`
      Playwright, которого у vitest нет).
- [ ] Осознанные исключения по `.skip` — через тот же baseline-механизм, что и храповик.
- [ ] Fail-closed: нечитаемый baseline исключений или недоступные исходники тестов →
      красный (аналог задачи фазы 3); чек секундный и стоит в начале fail-fast порядка.
- [ ] Тесты: PR с новым `.only` красит гейт; PR с новым `.skip` ведёт себя согласно
      зафиксированному режиму; исключение из baseline проходит зелёным.

**Когда готова:** PR с `it.only` красит unit-гейт; поведение по `.skip` соответствует
режиму, записанному до реализации.

### Фаза 5: Пост-мердж проверка деплоя

**Цель:** петля не строит следующую фазу поверх main, который не доехал до прода.
**Затрагивает:** ralph-раннер (`ralph.js`), нотифаер

**Задачи:**

- [ ] После статуса `merged` дожидаться итога deploy-workflow на смердженном sha
      (`gh run`, чтение с ретраями по образцу `ghJson`), таймаут ожидания — по замеру
      реального деплоя плюс запас.
- [ ] Healthcheck прода: HTTP 200 главной страницы (игровой смоук — в бэклог, не MVP).
- [ ] Красный итог (workflow failed / прод не отвечает) → пуш с указанием sha и итога
      workflow + стоп перед следующей фазой. Main раннер не трогает (alert-first).
- [ ] Только чтение: никаких мутаций, никакого авто-revert.
- [ ] Тесты: зелёный деплой не задерживает петлю дольше таймаута и не даёт ложного
      красного; упавший workflow даёт стоп + пуш; сетевой чих до прода не стопит петлю
      без ретраев.

**Когда готова:** после авто-мерджа с упавшим или откатившимся deploy-workflow раннер
не начинает следующую фазу (стоп + пуш); на эталонном зелёном мердже ложных красных нет.

### Фаза 6: Метрика пропущенных находок ревью

**Цель:** «ревью слабеет/крепнет» становится числом, а не ощущением.
**Затрагивает:** ralph-раннер, журнал в репозитории

**Задачи:**

- [ ] Считать находки ревью петли по severity из комментариев PR (`🔴/🟠/🟡/⚪` —
      ревью-промпт уже обязывает эту разметку, `gh` читает комментарии детерминированно).
- [ ] Журнал находок по фазам с разметкой источника «найдено ревью петли» / «найдено
      после»; формат и место (JSONL рядом с ralph-файлами либо в `docs/`) выбрать здесь.
- [ ] Дешёвый шаг фиксации ручной половины (команда или описанный ритуал утреннего
      разбора) — иначе журнал умрёт.
- [ ] Тесты: разбор комментариев по severity, запись в журнал, пустой набор находок.

**Когда готова:** по завершении фазы в журнале есть запись со счётом находок ревью по
severity, и задокументированный шаг фиксации «найдено после» существует и добавляет
запись с источником. Контрольная точка (риск PRD): если ручная половина не приживётся
за 2–3 фазы — журнал удаляем, а не верим кривому числу.

### Фаза 7: Research-вопросы в бэклог

**Цель:** отложенное не теряется — каждый research-вопрос виден на доске.
**Затрагивает:** бэклог GitHub

**Задачи:**

- [ ] Issue: мутационный смоук на критичных модулях (чем гонять, периодичность, бюджет).
- [ ] Issue: внешний systemd-сторож поверх взаимного контроля раннер↔монитор.
- [ ] Issue: игровой смоук прода (Playwright по живому сайту) поверх HTTP-healthcheck.
- [ ] Все — с метками `backlog`, `area:devops`, `complexity:*` и на доске проекта.
- [ ] Четвёртый research-вопрос критериев PRD — формат журнала находок — снимается
      фазой 6 и отдельным issue не заводится; если фаза 6 не сделана, он возвращается
      в бэклог.

**Когда готова:** на каждый research-вопрос PRD заведён issue со ссылкой на раздел PRD,
с метками и карточкой на доске.
