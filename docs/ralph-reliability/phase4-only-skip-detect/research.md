# Research: Детект `.only` / `.skip` — режим по `.skip`

**Дата:** 2026-07-22  
**Issue:** #159 «Зафиксировать режим по .skip до реализации»  
**PRD:** `docs/ralph-reliability/prd.md` (раздел 2, вектор 5, п. 2)

## Выбранный режим: вариант (а) — красный на любой новый skip

Гейт отвергает ВСЕ новые `it.skip` и `describe.skip` в unit-тестах. Осознанные исключения
идут через тот же baseline-механизм, что и храповик числа тестов.

### Обоснование

#### 1. Чистая точка отсчёта: нет наследия

В `src/` сейчас нет ни одного `.skip` (проверено grep). Это означает:

- При внедрении детекта мы не натыкаемся на наследие, требующее эскалации в baseline
- Новые `.skip` — явный выбор, не молчаливое копирование паттерна
- Точка отсчёта позволяет использовать режим без лимита и попытаться приучить команду к
  осознанному исключению тестов

#### 2. Строже версии (б) — но это осознанно

Красный на каждый `.skip` иногда бывает слишком строг: например, `.skip` под условием
платформы (`it.skip('windows only', ...)`) или за баг в том же PR должны иметь путь.

**Но этот путь существует:** baseline-механизм позволяет добавить исключение с `reason`,
и PR с обновлённым baseline проходит. Перемена инерции: вместо «спросить прощение» (silent
skip в ветке) — «получить одобрение» (обновить baseline в том же PR с аргументацией).

#### 3. Версия (б) — warning с лимитом — требует пороговых решений

«Warning, но не более N скипов в PR» — это добавляет новое свойство (порог), которое
нужно обосновывать и кодировать. На чистой точке отсчёта это преждевременно: видно ли
вообще нарастание скипов в реальных PR? Если нет, лимит убивает код без причины. Если да,
красный стоит правильнее.

### Механизм: baseline-исключения

Исключения из детекта `.skip` хранятся в файле:

```
scripts/skip-baseline.json
```

Формат (аналогичен `security-audit.baseline.json`):

```json
{
    "skips": [
        {
            "path": "src/features/game-engine/**/*.test.ts",
            "pattern": ".*platform-specific.*",
            "reason": "Скипы под платформу — вещь в себе, testing strategy знает о них",
            "expiresAt": "2026-08-05T00:00:00Z"
        }
    ]
}
```

`path` поддерживает только `**` и `*` — brace-expansion `{ts,tsx}`, `?` и классы `[...]`
`loadBaseline` отвергает явно (иначе они молча становятся литералами и исключение не
срабатывает).

**Правило:** PR, добавивший новый `.skip`, должен либо:

1. Обновить baseline с `reason` и `expiresAt`, либо
2. Сделать skip условным (код, а не декоратор), либо
3. Удалить `it.skip` и переписать тест чтобы не нужен был скип

**Fail-closed:** если baseline неправилен, `skip-baseline.json` недоступен или его парсинг
сломан — гейт краснеет. Нечитаемый JSON = стоп, не силентный пропуск.

#### TTL и остаточный риск чини-сессии

`expiresAt` обязателен у каждой записи (+14 дней от добавления, потолок 42 — тем же
механизмом `DEFAULT_TTL_DAYS`/`MAX_TTL_DAYS`, что и security-baseline). По истечении срока
гейт краснеет и выносит исключение человеку на пересмотр — исключение не живёт молча вечно.

**Остаточный риск (осознанно принят).** TTL не закрывает окно немедленно: чини-сессия
гейта (`gateHealAttempts`), которую не пускает красный `test:skip-detect`, может сама
дописать сюда исключение с формально валидным `reason`/`expiresAt` и пройти мердж без
повторного ревью (re-review в цикле есть только у `blocked`, не у gate-heal — тот же класс,
что инцидент 22.07 с security-baseline, #207). Полный барьер — детерминированный git-факт
«запись появилась после последнего ревью PR» → красный или принудительное re-review, по
образцу `baseline-policy.mjs`. Он выходит за рамки фазы 4 (это кросс-задача переиспользования
`baseline-policy` для не-security baseline, трек #155/#207) и здесь НЕ реализован. TTL —
интерим-барьер, который ограничивает срок жизни такого исключения, а не его появление.

### Варианты, отклонённые

#### (б) Warning с лимитом

**Отклонено.** На чистой точке отсчёта лимит выглядит произвольно; если проблема возникнет
(ползучее нарастание скипов), лимит всегда можно ввести потом. Сейчас красный честнее.

#### (в) Пассивный режим — логировать, не красить

**Отклонено.** Это выпускает джинна из бутылки: если чек молчалив, его будут игнорировать.
Громкие гейты — барьеры, работающие независимо от внимания.

## Следствия для других задач

- **Фаза 4.1:** Issue #160 (соседний в этой же фазе, чек it.only) использует ДРУГОЙ механизм:
  `.only` не имеет baseline, всегда красный. Обоснование: `.only` — отладочный конструкт, в
  `src/` ему не место; законного повода занести случайный `.only` в исключения нет, потому
  baseline ему не нужен.
- **Фаза 4.3:** baseline для `.skip` создаётся с пустым массивом, по мере нужды пополняется
  (как и security-baseline).
- **Доска:** нет отдельных research-issues; этот документ фиксирует единственный
  открытый вопрос PRD (п. 2, детект .skip) — выбор режима.
