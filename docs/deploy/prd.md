# PRD: deploy — прод-деплой на VDS с авто-TLS и авто-деплоем из GitHub

**Дата**: 2026-07-20
**Статус**: Ready — ключевые решения приняты (Caddy, systemd, GitHub Actions по SSH, билд на VDS, healthcheck+автооткат, cron-бэкап). Готов к разбивке на фазы.

## Цель

Сделать Pixel Tanks публично доступным по адресу `https://pixeltanks.ru` с валидным TLS-сертификатом и настроить автоматический деплой: любой push в ветку `main` без ручных действий обновляет прод. Сейчас приложение живёт только локально — цель фазы деплоя вывести его в интернет так, чтобы дальнейшие релизы стоили «git push» и ничего больше.

> Мини-глоссарий (для Димы — серверные концепты в одну фразу):
>
> - **VDS** — арендованный виртуальный сервер в облаке, всегда включён, имеет свой публичный IP.
> - **DNS A-запись** — строчка в «телефонной книге интернета»: доменное имя → IP-адрес сервера. **AAAA** — то же самое, но для IPv6-адреса.
> - **Reverse-proxy** — программа-«вахтёр» перед приложением: принимает запросы из интернета на портах 80/443 и передаёт их внутрь на локальный порт приложения.
> - **TLS-сертификат** — «паспорт сайта», делает `https://` и замочек в браузере; выдаётся бесплатно центром Let's Encrypt, живёт 90 дней, надо продлевать.
> - **systemd-юнит** — штатный «менеджер процессов» Linux: запускает приложение при старте сервера, перезапускает при падении, ведёт логи.
> - **GitHub Actions** — CI/CD-робот GitHub: по событию (push) выполняет заданный сценарий на своих раннерах.

## Пользовательские сценарии

- Игрок открывает `https://pixeltanks.ru` в браузере → видит приложение Pixel Tanks по HTTPS с валидным сертификатом (замок, без предупреждений).
- Игрок открывает `http://pixeltanks.ru` (без https) → автоматически редиректится на `https://`.
- Разработчик (Дима) мержит PR в `main` → через несколько минут прод сам обновляется до нового кода, без SSH-захода руками.
- Сертификат подходит к концу срока (90 дней) → продлевается автоматически, сайт не «краснеет».
- Сервер перезагрузился (reboot, обновление ОС) → приложение и reverse-proxy сами поднимаются, сайт снова доступен без ручного вмешательства.
- Соседние процессы на VDS (ralph-loop, Shadowsocks-туннель) продолжают работать → деплой прод-приложения их не задевает.

## В скоупе

1. **DNS**: A-запись `pixeltanks.ru` → `186.246.7.204`, AAAA-запись `pixeltanks.ru` → `2a03:6f00:a::2:d005`, аналогичные записи для `www.pixeltanks.ru` (www редиректит на apex-домен). Управление — через Timeweb (домен на этом же аккаунте, сейчас `linked_ip: null`).
2. **Reverse-proxy Caddy**: терминирует HTTPS на `:443`, слушает `:80` (редирект на https), автоматически выпускает и продлевает TLS-сертификат Let's Encrypt для `pixeltanks.ru` / `www.pixeltanks.ru`, проксирует трафик на локальный порт приложения. Caddyfile — минимальный, декларативный.
3. **Прод-порт приложения**: Next.js (`npm run start`) слушает **`:3060`** локально (не конфликтует с dev `:3050`; порт 3060 никуда наружу не открыт, доступен только Caddy на localhost).
4. **Отдельный прод-клон репозитория**: прод разворачивается из выделенного git-каталога (напр. `/opt/pixel-tanks` или `/srv/pixel-tanks`), **отдельного** от рабочей копии ralph-loop — чтобы git-операции ralph не ломали прод.
5. **systemd-юнит приложения**: рабочий каталог = прод-клон; `ExecStart` = запуск `next start` на порту 3060; `Restart=always`; `EnvironmentFile` с секретами (`PAYLOAD_SECRET`, `DATABASE_URI`, `NODE_ENV=production`, `PORT=3060`); `systemctl enable` — автозапуск после reboot.
6. **Deploy-пользователь**: отдельный системный пользователь с минимальными правами (не root), владелец прод-каталога; его публичный SSH-ключ — в `authorized_keys`; ограниченный `sudo` только на `systemctl restart` прод-юнита (и, при необходимости, reload Caddy) — ничего больше.
7. **GitHub Actions workflow (авто-деплой)**: на `push` в `main` раннер по SSH заходит на VDS под deploy-пользователем и в прод-каталоге выполняет `git pull` → `npm ci` → `npm run build` → `sudo systemctl restart` прод-юнита → **healthcheck** (см. п.13). Приватный deploy-ключ — в GitHub repo secrets, публичный — в `authorized_keys` на VDS.
8. **База данных**: SQLite-файл (`payload.db`) лежит в стабильном каталоге на диске VDS **вне прод-клона git** (или в каталоге, который `git pull`/`npm ci`/build не перетирают), путь задаётся через `DATABASE_URI` в EnvironmentFile. Файл переживает деплои.
9. **Firewall (ufw)**: открыты только `22` (SSH), `80` (http→https редирект), `443` (https). Прод-порт `3060` наружу закрыт.
10. **Swap-файл**: создать swap-файл **2–4 GB** и включить (`swapon`, запись в `/etc/fstab` для переживания reboot). Причина: сейчас swap = 0; замер показал пик билда ~1.7 GB, запас есть, но без swap любой неожиданный пик = мгновенный OOM-killer вместо просадки. Дешёвая страховка на диске 80 GB (занято ~7).
11. **Healthcheck + автооткат деплоя**: после `systemctl restart` workflow дёргает `https://pixeltanks.ru` (или локальный `127.0.0.1:3060/`); если ответ не `200` в течение таймаута — автоматически откат на предыдущий git-коммит (`git reset --hard HEAD@{1}` / previous SHA) + `npm ci` + `build` + restart, и workflow падает с ошибкой (виден в Actions). Прод не остаётся лежать из-за плохого коммита.
12. **Cron-бэкап `payload.db`**: ежедневный дамп файла БД в отдельный каталог на диске VDS (напр. `/var/backups/pixel-tanks/`) с ротацией (хранить последние N дней, напр. 7). Спасает от кривого деплоя/миграции. Offsite (S3) — будущее, см. non-goals.
13. **Payload admin (`/admin`)**: доступен публично, вход по штатному email/паролю Payload (local auth). Дополнительных слоёв (basic-auth, IP-allowlist) на этой итерации не ставим. Rate-limit на форму логина — отметить как будущее усиление.

## Не в скоупе

- CDN / кэширующий edge перед сайтом — будущее.
- Горизонтальное масштабирование, несколько инстансов, балансировщик — не нужно на текущем трафике.
- Отдельное staging / preview-окружение — пока деплоим сразу в прод из `main`.
- Миграция с SQLite на PostgreSQL (`@payloadcms/db-postgres`) — будущая фаза, когда потребуется.
- Мониторинг / алертинг / uptime-проверки (Grafana, healthcheck-пинги) — будущее.
- **Offsite-бэкап** `payload.db` (S3 Timeweb / другой хост) — будущее. На этой итерации бэкап только на диск того же VDS (cron, см. «В скоупе») — не спасает от гибели самого диска.
- Docker / контейнеризация прод-приложения — сознательно не используем (systemd + прямой Node).
- Zero-downtime / blue-green деплой — рестарт юнита даёт короткий (секунды) простой, это приемлемо.
- Дополнительная защита `/admin` (basic-auth, IP-allowlist, rate-limit логина) — на этой итерации только штатный логин Payload; усиление — будущее.

## Технические ограничения

- **Соседство на одном VDS** (`ralph-prod`, 4 CPU / 8 GB / 80 GB, занято ~7 GB). На сервере уже работают: автономный ralph-loop (делает git-операции в СВОЕЙ рабочей копии) и Shadowsocks-туннель к Anthropic (systemd). Прод-приложение живёт рядом и **не должен** их задевать:
    - Прод — из отдельного git-клона, не из копии ralph.
    - Прод-порт `3060` не конфликтует с dev `3050` и портами туннеля.
    - Ресурсы делятся: `npm run build` (Next.js) — тяжёлая операция по CPU/RAM. **Замерено на этом VDS** (2026-07-20): пик билда `Maximum resident set size ≈ 1.7 GB`, время ~37 с. При свободных ~5 GB (даже с запущенным VS Code Server) запас двукратный → **решение: собираем на VDS**. Митигация риска пиков — swap-файл (см. «В скоупе»).
- **Порты провайдера**: `80` и `443` НЕ в блок-листе Timeweb → веб-трафик разрешён. `22` открыт. Заблокированы `25/465/587/2525/53413/3389/389/5060` — почтовые/прочие порты приложению не нужны.
- **Стек приложения**: Next.js 16 + React 19 + Payload CMS 3 + SQLite (`@payloadcms/db-sqlite`), Node 24. `next start` по умолчанию слушает `:3000` — переопределяем на `3060` через `PORT`.
- **Домен**: `pixeltanks.ru` на аккаунте Timeweb, оплачен до 2027-07-19, статус paid, provider TimewebVirtreg. Сейчас `linked_ip: null` — записи нужно создать.
- **TLS через Let's Encrypt** требует, чтобы DNS уже указывал на сервер и порты 80/443 были доступны из интернета (ACME HTTP-challenge) — DNS настраиваем ДО запуска Caddy, иначе выпуск сертификата не пройдёт.
- **Секреты вне git**: `PAYLOAD_SECRET`, `DATABASE_URI`, deploy-SSH-ключ — только в systemd `EnvironmentFile` и GitHub secrets, никогда в репозитории. `payload.db` не коммитить (уже правило проекта).
- **Выбор «GitHub Actions по SSH» vs альтернативы** (зафиксировано): деплой push-моделью (GitHub Actions → SSH → pull/build/restart) выбран против pull-таймера (cron `git pull` на сервере) и вебхука. Плюсы: стандарт индустрии, мгновенный деплой сразу после мержа, логи каждого деплоя видны в GitHub, единый источник триггера. Минус/цена: требуется хранить приватный deploy-ключ в GitHub secrets (принимаем, ключ — выделенный, с минимальными правами).

```
   │     │  reverse-proxy на localhost
   │     ▼
   ├─ Next.js (systemd-юнит)  127.0.0.1:3060  ──►  Payload CMS
   │                                                   │
   │                                                   ▼

   │

   └─ [сосед] Shadowsocks-туннель (systemd) — не трогается
```

## Поток деплоя

```
Дима мержит PR → push в main (AtlantaTeam/pixel-tanks)
   │
   ▼
GitHub Actions (workflow on: push, branch: main)
   │  SSH под deploy-пользователем (приватный ключ из repo secrets)
   ▼
VDS: прод-каталог (/opt/pixel-tanks)
   ├─ git pull                     (подтянуть новый код)
   ├─ npm ci                       (детерминированная установка зависимостей)
   ├─ npm run build                (сборка Next.js — пик ~1.7 GB, ~37 с)
   ├─ sudo systemctl restart pixel-tanks   (ограниченный sudo — только этот юнит)
   └─ healthcheck https://pixeltanks.ru
         ├─ 200 OK  → деплой успешен ✓
         └─ не 200  → АВТООТКАТ на предыдущий коммит (git reset + ci + build + restart),
                      workflow падает с ошибкой (виден в Actions)
   │
   ▼
Приложение поднимается на :3060 → Caddy сразу отдаёт новую версию по https
```

## Пошаговые требования к настройке

Ниже — ЧТО должно быть сделано (требования), не готовые конфиги/команды.

1. **DNS** (Timeweb): создать A `pixeltanks.ru` → `186.246.7.204`, AAAA `pixeltanks.ru` → `2a03:6f00:a::2:d005`, аналогично для `www`; дождаться распространения (проверяемо `dig`/`nslookup`).
2. **Firewall (ufw)**: разрешить только `22/80/443`, включить ufw. Убедиться, что `3060` наружу закрыт.
3. **Прод-каталог и git-клон**: создать выделенный каталог (напр. `/opt/pixel-tanks`), клонировать `AtlantaTeam/pixel-tanks` (ветка `main`), владелец — deploy-пользователь. Каталог физически отдельный от рабочей копии ralph.
4. **Каталог БД**: определить стабильный путь для `payload.db` вне git-клона (или защищённый от перетирания при деплое); прописать его в `DATABASE_URI`.
5. **EnvironmentFile с секретами**: файл вне репозитория (напр. `/etc/pixel-tanks/env` с правами `600`), содержит `PAYLOAD_SECRET`, `DATABASE_URI`, `NODE_ENV=production`, `PORT=3060`.
6. **systemd-юнит приложения** (`pixel-tanks.service`): `WorkingDirectory` = прод-клон, `ExecStart` = `next start` (через `npm run start`/node), `EnvironmentFile` = файл из п.5, `Restart=always`, запуск под deploy-пользователем; `systemctl enable` для автозапуска после reboot.
7. **Установка Caddy** как systemd-сервиса; `Caddyfile` с требованиями: домены `pixeltanks.ru` и `www.pixeltanks.ru`, авто-TLS Let's Encrypt, `www` → редирект на apex, `http` → редирект на `https`, `reverse_proxy` на `127.0.0.1:3060`. Caddy тоже `enable` для автозапуска.
8. **Deploy-пользователь**: создать системного пользователя без root, владельца прод-каталога; настроить `sudoers`-правило, разрешающее ТОЛЬКО `systemctl restart pixel-tanks` (и при необходимости `systemctl reload caddy`) без пароля; больше никаких sudo-прав.
9. **SSH-ключ деплоя**: сгенерировать выделенную пару ключей; публичный — в `authorized_keys` deploy-пользователя; приватный — в GitHub repo secrets (напр. `DEPLOY_SSH_KEY`, плюс `DEPLOY_HOST`, `DEPLOY_USER`).
10. **GitHub Actions workflow**: `.github/workflows/deploy.yml`, триггер `push` в `main`; шаг — SSH на VDS и выполнение `git pull && npm ci && npm run build && sudo systemctl restart pixel-tanks` в прод-каталоге. Логи деплоя видны во вкладке Actions.
11. **Ротация секретов**: `PAYLOAD_SECRET` / `DATABASE_URI`, ранее прошедшие через чат, — сгенерировать заново перед публичным запуском (см. open question проекта). Прод-значения не должны совпадать с «засвеченными».
12. **Swap-файл**: создать swap-файл 2–4 GB, включить, прописать в `/etc/fstab` (переживает reboot). Сделать ДО первого прод-билда.
13. **Healthcheck в workflow**: после restart — HTTP-проверка (ожидаем `200`); при провале — автооткат на предыдущий коммит (reset + ci + build + restart) и падение workflow с ненулевым кодом.
14. **Cron-бэкап БД**: настроить ежедневный дамп `payload.db` в `/var/backups/pixel-tanks/` с ротацией N дней (напр. через systemd-timer или cron + скрипт с `sqlite3 .backup`).

## Секьюрити-требования

- **Секреты только вне репозитория**: `PAYLOAD_SECRET`, `DATABASE_URI`, приватный deploy-ключ — в systemd `EnvironmentFile` (права `600`) и GitHub secrets. Ничего из этого не попадает в git (проверяемо grep'ом по истории/рабочему дереву).
- **Минимальные права deploy-пользователя**: не root; ограниченный `sudo` строго на рестарт нужного юнита (и reload Caddy), выданный через отдельное `sudoers`-правило. Никакого общего `sudo`.
- **Отдельный SSH-ключ под деплой**: используется только CI, не совпадает с личным ключом Димы; при компрометации отзывается независимо (удаление из `authorized_keys`).
- **Firewall**: наружу открыты только `22/80/443`. Прод-порт `3060` слушает только `127.0.0.1`, наружу недоступен.
- **Ротация «засвеченных» секретов**: значения, ранее фигурировавшие в переписке, считаются скомпрометированными — для прода генерируются новые.
- **Изоляция от соседей**: прод-процесс и его файлы (каталог, БД, env) принадлежат deploy-пользователю; ralph-loop работает под своим пользователем/каталогом и не имеет доступа на запись к прод-артефактам.

## Риски и открытые вопросы

Решения ниже приняты Димой (ЛПР) 2026-07-20; оставшиеся открытые вопросы помечены явно.

- **Соседство с ralph-loop**: прод и ralph делят один VDS и ресурсы. Риск — ralph случайно затронет прод-каталог/порт. Митигация: жёсткая изоляция (отдельный каталог, пользователь, порт). **Открытый вопрос (фактический, выясняется на сервере)**: под каким именно пользователем/путём живёт ralph сейчас — надо посмотреть перед созданием прод-каталога, чтобы гарантированно развести. (На момент замера рабочая копия ralph — `/root/pixel-tanks`; прод пойдёт в `/opt/pixel-tanks` под отдельным пользователем.)
- **Деплой во время активной сборки / нагрузки** — ✅ РЕШЕНО: собираем на VDS. Замер: пик билда ~1.7 GB при ~5 GB свободных (запас 2×). Митигация пиков — swap-файл. Копирование `.next` с раннера отклонено как усложнение.
- **Откат неудачного деплоя** — ✅ РЕШЕНО: healthcheck после restart + автооткат на предыдущий коммит (в скоупе, п.11/13).
