# Research

Проведи технический research для плана: $ARGUMENTS

Research делается ОДИН раз на весь план, не на каждую задачу. Цель — принять все технические решения ДО реализации, чтобы агент (в т.ч. автономный Ralph Loop) не выбирал библиотеки наугад и не останавливался на вопросах.

## Порядок действий

1. Прочитай план и связанный PRD.
2. Выпиши все места, где есть выбор: библиотеки, API, архитектурные подходы, форматы хранения.
3. Для каждого выбора: изучи 2-3 варианта С УЧЁТОМ стека проекта (см. CLAUDE.md, package.json). Вариант, несовместимый со стеком (например, Express-middleware в Fastify-проекте), отбрасывай сразу.
4. Прими решение по каждому пункту и обоснуй в 1-2 предложения.

## Результат

Сохрани отчёт в `docs/{feature}/research.md` — рядом с prd.md и plan.md, в папке той же фичи:

```markdown
# Research: {название фичи}

**План:** {ссылка}
**Дата:** {дата}

## Решения

### {Вопрос 1: например, загрузка файлов}

**Выбрано:** {библиотека/подход}
**Альтернативы:** {что рассматривалось и почему отклонено — кратко}
**Как применять:** {ключевые моменты интеграции, 3-5 строк}
```

## После сохранения

Добавь в CLAUDE.md проекта (раздел с документацией) строку-ссылку на отчёт, чтобы агент обращался к нему при реализации.

## Правила

- Все решения принимаются ЗДЕСЬ — при реализации вопросов остаться не должно.
- Учитывай версии зависимостей проекта (совместимость).
- Пиши компактно: отчёт читается агентом в каждой сессии реализации, не раздувай токены.
