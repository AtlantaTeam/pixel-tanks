# PRD генератор

Создай PRD (Product Requirements Document) для фичи: $ARGUMENTS

Сохрани в `docs/{feature}/prd.md` (название фичи переведи на английский, kebab-case; у каждой фичи своя папка — там же потом появятся plan.md и research.md). Если папки нет — создай.

## Структура документа

```markdown
# PRD: {название фичи}

**Дата**: {текущая дата}
**Статус**: Draft

## Цель

Одно-два предложения: что и зачем это нужно пользователю.

## Пользовательские сценарии

- Пользователь {действие} → {результат}

## В скоупе

Конкретный список того, что входит в фичу.

## Не в скоупе

Что явно НЕ делаем в этой итерации.

## Технические ограничения

Известные ограничения, которые нужно учесть (стек, интеграции, производительность).

## Критерии готовности

- [ ] Критерий 1 (проверяемый)
- [ ] Критерий 2
```

## Правила

- Пиши конкретно, без воды.
- Критерии готовности — только проверяемые (можно однозначно ответить да/нет).
- Не описывай КАК реализовывать — только ЧТО и ЗАЧЕМ.
- Если описание фичи слишком короткое или неоднозначное — задай уточняющие вопросы ДО создания файла, пока не будет полного понимания.

## После создания PRD

Когда PRD согласован с пользователем (внесены его правки, возражений нет) — запусти скилл `plan-phase` для этой фичи; тот, в свою очередь, автоматически заведёт milestones/issues и доску через скилл `issues`. Цепочка prd → plan-phase → issues идёт без отдельных команд пользователя.
