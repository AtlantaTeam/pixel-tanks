# Ralph Runbook — операционная процедура

Как запускать автономный раннер, что проверить до старта и на какие грабли уже наступали.
Это операционный документ для человека и для агента-помощника; правила для кодер-сессии
внутри петли — в `ralph.md`, они про другое.

## Перед стартом фазы

1. **Конфиг фазы обязан быть в `origin/main`.** Раннер детачит своё дерево на
   `origin/main`, локальных правок он не увидит вовсе:

    ```bash
    # .claude/ralph/ralph.config.json → common.phases[0]
    #   milestone: точное название milestone фазы
    #   branch:    ветка фазы (feature/…)
    ```

    Везти отдельной веткой и отдельным PR, **не смешивая с docs**: 21.07 незакоммиченный
    конфиг фазы 5 уехал в чужой docs-PR (#144), потому что ветку резали от локального main.

2. **Milestone существует, issues размечены.** Без `complexity:*` роутинг моделей не
   видит карточку (правило #142). Проверить:

    ```bash
    gh issue list --milestone "<название фазы>" --state open --json number,title,labels
    ```

3. **Секреты на месте.** Профиль `prod` требует в `/root/ralph.env` именно
   `RALPH_TG_BOT_TOKEN` и `RALPH_TG_CHAT_ID` (не `TG_*` — на этом сработал fail-closed
   22.07: без пушей прод-режим стартовать отказывается, и это правильно).

4. **Раннер не запущен вторым экземпляром.** Лока пока нет (#176–#179), проверять руками:

    ```bash
    pgrep -af "ralph.js|monitor.js"
    ```

## Запуск

Раскладка: **отдельные окна tmux**, `2: ralph` — цикл, `3: monitor` — дашборд.

```bash
# окно 2 — цикл
tmux new-window -t work -n ralph -c /root/pixel-tanks
tmux send-keys -t work:2 'set -a && . /root/ralph.env && set +a && node .claude/ralph/ralph.js --profile prod' Enter

# окно 3 — дашборд (фаза, submitted, прогресс issues, открытые PR, порог тишины)
tmux new-window -t work -n monitor -c /root/pixel-tanks-ralph
tmux send-keys -t work:3 'node .claude/ralph/monitor.js --profile prod --interval 60 --config /root/pixel-tanks/.claude/ralph/ralph.config.json' Enter
```

Монитор запускать **из дерева раннера** (`/root/pixel-tanks-ralph`): он берёт
`ralph.log` и state от `__dirname`. `--config` — абсолютный путь конфига из дерева
человека, тот самый файл, по которому идёт раннер. Второй экземпляр монитора безопасен:
он read-only и pid-файла не трогает (раннер поднимает свой detached-экземпляр сам).

**`--reset`** сбрасывает state на первую фазу конфига **и выходит** — это не запуск.
Нужен, когда state пустой или указывает на прошлую фазу; после него запускать обычной
командой.

## Профили

| Профиль      | Гейт                                   | Пуши       | После мерджа             |
| ------------ | -------------------------------------- | ---------- | ------------------------ |
| `playground` | build, lint, lint:fsd, typecheck, test | только лог | сразу следующая фаза     |
| `prod`       | то же + security, coverage, e2e        | в Telegram | стоп перед деплоем (#87) |

С фазы 3 надёжности (#216–#219) `prod` разбирает blocked сам: правки → повторное ревью
моделью не слабее поставившей блок → метку снимает раннер по вердикту. К человеку уходит
только тупик — три ревью подряд, оставившие блок. Человеческий стоп-кран теперь отдельный:
метка **`hold`** проверяется первой, сильнее `blocked`, и раннер её не снимает ни на одном
пути — снять можно только руками (`gh pr edit <N> --remove-label hold`).

## Красный пост-мердж деплой (#163–#167, prod)

С фазы 5 наблюдаемости раннер после мерджа prod-фазы **дожидается итога deploy-workflow**
на смердженном sha и делает **healthcheck прода** (HTTP 200 главной). Если workflow упал/не
досмотрен или прод не ответил — раннер ставит **барьер** `state.deployBlock`, шлёт пуш `⛔` и
останавливается: **следующая фаза не начнётся**, пока барьер не снят. Проверка строго
read-only — раннер `main` и прод не трогает.

Ночной сценарий «пришёл пуш ⛔ про красный деплой»:

1. **Смотри run деплоя** — ссылка в пуше и в `state.deployBlock.url`
   (`.claude/ralph/ralph.state.json`); поле `reason` говорит, что именно красное
   (`workflow failure` / `timeout` / `прод не отвечает` / pending — проверка не досмотрена,
   процесс мог умереть в окне ожидания).
2. **Откат релиза — за deploy-workflow** (`.github/workflows/deploy.yml`), не за раннером:
   `main` он не трогает. Почини прод/деплой обычным путём.
3. **Продолжить после починки** — перезапуск loop с флагом **`--deploy-resolved`**: он
   снимает барьер (единственный способ, как и `hold` — осознанное решение человека, раннер
   сам `deployBlock` не снимает). Барьер допушивается ещё раз на старте, если прошлый прогон
   умер между записью барьера и пушем.
4. **`--reset` при активном барьере отказывает** без `--deploy-resolved` — чтобы сброс state
   по несвязанной причине молча не стёр красный барьер.

## Правки человека в ветку фазы

Если разбираешь PR фазы и правишь его сам, помни про инвариант H3: гейт мерджит, только
когда локальный ref ветки совпадает с головой PR. Иначе — честный стоп с сообщением
«в main уехал бы не тот код, что лежит локально».

- **Правишь в дереве раннера** (`/root/pixel-tanks-ralph`) — только когда прогон
  остановлен, иначе гонка с чини-сессией за `.next`/`node_modules`.
- **Правишь в своём дереве** — после `git push origin HEAD:<ветка>` обязательно
  синхронизируй локальный ref, иначе гейт встанет:

    ```bash
    git fetch origin <ветка>:<ветка>
    ```

Наступал 22.07: закоммитил фикс baseline из своего дерева, запушил — гейт отказался
мерджить, потому что ref ветки в репозитории раннера остался на старом sha.

## Грабли (каждая — из реального прогона)

- **Панели tmux нумеруются с 1.** `work:2.1` — это ПЕРВАЯ панель, то есть сам раннер;
  ввод туда уходит в stdin работающего процесса. Панель монитора — `work:2.2`.
- **`tail -f` не отдаёт панель по одному Ctrl+C** и молча съедает последующий ввод:
  окно выглядит рабочим, а команды не исполняются.
- **Окна переиндексируются** при `kill-window` (renumber-windows). После удаления окна 2
  монитор из 3 переезжает в 2 — проверять `tmux list-windows` перед `send-keys`, иначе
  команда уходит не туда.
- **Выход из copy-mode: `q` может прилипнуть к следующей команде** (`qset -a && …`).
  Проверять `#{pane_in_mode}` вместо слепой отправки `q`.
- **Во время прогона не трогать конфиг и milestone'ы.** Конфиг читается один раз в
  `main()`, правка milestone на лету сбивает выборку issue и подсчёт остатка фазы.
- **Не запускать команды в дереве раннера во время прогона** (`npm run build` и т.п.):
  это гонка с чини-сессией за `.next` и `node_modules`.
- **Вкладка `ralph` во время сессии выглядит мёртвой** — `claude -p` печатает вывод
  только в конце. Судить о жизни по `ralph.log`, дашборду и (с фазы 1) по deadman-пушу.

## Что смотреть во время прогона

```bash
tail -f /root/pixel-tanks-ralph/.claude/ralph/ralph.log    # хореография
gh pr list --head <ветка фазы>                              # PR фазы
gh issue list --milestone "<фаза>" --state open             # остаток
```

Ключевые маркеры лога: `🔄 итерация`, `🔺 Ревью эскалировано`, `🚦 Гейт`,
`✗ <чек> — красный`, `🩹 Чини-сессия`, `✅ PR #N смерджен`, `🏁 Milestone закрыт`,
`⏸ loop остановлен перед деплоем`, `⛔` — любой fail-closed стоп.

## После фазы

- `prod` останавливает петлю после мерджа (стоп перед деплоем) — следующая фаза требует
  нового запуска. Перед стопом раннер дожидается итога деплоя и healthcheck прода; красный
  итог ставит барьер (см. «Красный пост-мердж деплой»), снимаемый только `--deploy-resolved`.
- Доска синхронизируется сама (`scripts/project-sync.mjs`, #199), milestone закрывается
  раннером. Если что-то разъехалось — `npm run project:sync`.
- Ревью фазы живёт в комментариях PR. Если ревью запускалось из сессии-помощника —
  публиковать его в PR, а не оставлять в контексте: иначе для команды его не существует.
