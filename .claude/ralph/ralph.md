# Ralph Loop — правила автономной работы

Ты работаешь в автономном цикле: одна сессия = один Issue. Внешний раннер сам запустит следующую сессию.

## Как брать Issue

- Возьми ПЕРВЫЙ открытый Issue из указанного milestone (наименьший номер).
- Прочитай title, body и критерии готовности целиком.
- Работай только в указанной ветке. Сначала `git fetch origin main` и проверь, есть ли ветка на origin (`git ls-remote --exit-code --heads origin <ветка>`):
    - **Ветка есть на origin** (прошлые сессии её уже запушили, свежий worktree/новая машина — локального ref может не быть) — возьми её: `git checkout -b <ветка> origin/<ветка>` (или `git checkout <ветка>`, если ref уже локально). НЕ создавай её заново от `origin/main` — получишь расходящуюся ветку, пуш отвергнется non-ff, и до force-push доходить нельзя.
    - **Ветки нет нигде** — только тогда создай от свежего origin/main: `git checkout -b <ветка> origin/main`.
- НЕ ветвись от локального `main`: раннер живёт в выделенном worktree, локальный `main` держит дерево человека и он может отставать от origin. Новых веток не создавай.

## Реализация

- Сначала тесты, потом реализация (TDD): red → green → refactor.
- После каждого значимого изменения запускай тесты (`npm run test --silent`).
- Используй существующие конвенции проекта (см. CLAUDE.md и .claude/rules/).
- Если для задачи есть research-документ (`docs/{feature}/research.md`) — следуй принятым там решениям, не выбирай библиотеки заново.

## UI-задачи: визуальный контроль (обязательно)

Если Issue затрагивает то, как игра выглядит или управляется (layout, HUD, тач, анимации, страницы):

- ПЕРЕД вёрсткой запусти скилл `ui-ux-pro-max` (глобальный, `~/.claude/skills/ui-ux-pro-max/`): `python <skill>/scripts/search.py "<запрос>" --domain ux` (или `--design-system`) — возьми оттуда чек-лист под задачу. Наша тема: retro pixel game (Pixel Art, Press Start 2P, тёмный фон).
- ПОСЛЕ реализации проверь визуально через Playwright MCP: `npm run dev` (порт 3050) → навигация → скриншот в `screenshots/` → `browser_console_messages` (ошибок быть не должно). Мобильные Issue — проверять на вьюпортах 375×667 и 667×375.
- Не закрывай Issue без визуальной проверки — «код написан» ≠ «выглядит правильно».

## Circuit breaker (обязательно)

- Если тесты красные после **5 попыток** исправления — остановись, оставь комментарий в Issue с описанием проблемы и того, что пробовал, и заверши сессию. НЕ закрывай Issue.
- Если задача требует решения, которого нет ни в Issue, ни в PRD, ни в research — оставь вопрос комментарием в Issue и заверши сессию.
- Если упёрся в ручной гейт (например, `npm install` требует подтверждения человека): повесь на Issue label `blocked` (`gh issue edit N --add-label blocked`), опиши комментарием, что именно нужно от человека, и заверши сессию. Раннер пропустит заблокированный Issue и возьмёт следующий; label снимет человек после разблокировки. Важно: пока в milestone висит хоть один открытый blocked-Issue, раннер НЕ сдаёт фазу (PR/мердж отложены до разбора человеком) — «очередь пуста» не значит «фаза готова».

## Если заводишь issue сам

Иногда по ходу работы (или разбирая комментарии ревью) находится задача вне текущего скоупа. Заводить под неё issue — правильно, но голая карточка теряется: без меток её не видит роутинг моделей, без доски — человек.

- Метки обязательны: `complexity:*` (по трудоёмкости — от неё зависит, какой моделью раннер возьмёт задачу), `area:*` (зона) и `backlog`, если задача вне текущей фазы.
- Добавь на доску: `gh project item-add 1 --owner AtlantaTeam --url <ссылка на issue>`. Бэклог живёт там, issue мимо доски равно потерянному.
- Milestone текущей фазы НЕ ставь: раннер выбирает issue по milestone и считает по нему остаток фазы — новая карточка удлинит фазу, которую ты сейчас сдаёшь. Отложенное — в бэклог, человек разложит по фазам.
- В теле — симптом, причина и проверяемый критерий готовности, а не только «надо бы поправить». Issue #140 завёлся без меток и мимо доски именно потому, что этого правила здесь не было.

## Завершение

- Убедись: все тесты зелёные, все критерии готовности Issue выполнены.
- Прогони `npm run build` — гейт мерджа проверяет и сборку; tsc и тесты не ловят ошибки Turbopack/RSC (например, `'use server'`-модуль может экспортировать только async-функции).
- Коммить по conventional commits (см. скилл git-commit). Коммит должен ссылаться на Issue (`#N`).
- Ключевые слова автозакрытия — **только английские**: `Closes #N` / `Fixes #N` / `Resolves #N`. Русское «Закрывает #N» GitHub игнорирует, и после мерджа issue остаётся открытым (проверено на #130). Само описание PR при этом на русском, как и везде.
- Запушь рабочую ветку в origin (`git push -u origin <ветка>`) — работа фазы не должна жить только на локальной машине: гейт мерджа сверяет локальный HEAD с головой PR, незапушенные коммиты остановят авто-мердж.
- Закрой Issue с коротким комментарием: что сделано, какие файлы затронуты.
- НЕ создавай PR — это сделает раннер после завершения всех issues фазы.
- После закрытия Issue немедленно заверши сессию. НЕ бери следующий Issue — раннер запустит новую чистую сессию.
